## 目的

アプリケーションの変更を安全に本番環境へリリースする。
デプロイ前チェック・マイグレーション実行・アプリケーションデプロイ・ヘルスチェック・
異常時ロールバックの一連のプロセスを実施し、リリースノートを生成する。

## 実行プロセス

### Step 1: デプロイ前チェック

**実行内容**:
- デプロイ対象のブランチ・タグ・コミットを確認する
- CIパイプラインの全テストがパスしているかを確認する
- セキュリティスキャン・脆弱性チェック結果を確認する
- 統合テストレポートのGo/No-Go判定を確認する
- マイグレーションがある場合は `database-migration-design` スキルが生成した設計書（通常 `docs/migration-design.md`）を確認する。なければスキップする
- 本番DB・インフラの現在の状態（ヘルス）を確認する
- リリースウィンドウの確認と関係者への通知を確認する
- ロールバック手順の最終確認を行う
- オンコール担当者の確認を行う

**専門性の要件**: DevOps、リリース管理

**期待成果物**: デプロイ前チェックリスト（全項目OK/NG状態付き）

**完了基準**: 全チェック項目がOKになっている（NGがある場合はデプロイ中止）

### Step 2: デプロイ環境設定確認

**実行内容**:
- 環境変数・シークレットの設定状況を確認する（機密値は確認のみ、値は表示しない）
- インフラリソース（CPU、メモリ、ディスク）の空き容量を確認する
- デプロイ先の設定ファイル（Kubernetes manifest, terraform状態等）を確認する
- 依存サービス（DB、キャッシュ、外部API）の接続確認を行う
- 必要なIAMロール・権限が設定されているかを確認する

**専門性の要件**: インフラ管理、クラウドプロバイダー

**期待成果物**: 環境設定確認レポート

**完了基準**: 全依存サービスへの接続が確認され、環境設定に問題がない

### Step 3: データベースマイグレーション実行

**前提条件**: デプロイするDBマイグレーションがある場合のみ実行する。マイグレーションがない場合はこのステップをスキップして Step 4 に進む。

**実行内容**:
- マイグレーション実施前にDBバックアップを取得する（取得完了を確認する）
- マイグレーションをドライラン（--dry-run）で事前確認する（対応している場合）
- UPマイグレーションを実行する:
  ```bash
  # 例: Alembic
  alembic upgrade head
  # 例: Rails
  rails db:migrate
  # 例: Flyway
  flyway migrate
  ```
- マイグレーション実行後にスキーマの整合性を確認する
- データ変換が含まれる場合は変換後のデータを検証する
- マイグレーション失敗時は即座にロールバックを実施する

**専門性の要件**: データベース管理、マイグレーション実行

**期待成果物**: マイグレーション実行ログと検証結果

**完了基準**: マイグレーションが正常完了し、スキーマとデータの整合性が確認されている

### Step 4: アプリケーションデプロイ

**実行内容**:
- デプロイ戦略に従ってアプリケーションをデプロイする:
  - **ローリングデプロイ**: 段階的にポッドを入れ替える
    ```bash
    kubectl set image deployment/app app=image:tag
    kubectl rollout status deployment/app
    ```
  - **ブルーグリーンデプロイ**: 新環境を起動し、トラフィックを切り替える
  - **カナリアリリース**: 一部トラフィックのみ新バージョンに転送する
- デプロイ進行状況をモニタリングする
- 各レプリカの起動確認を行う
- デプロイ完了後の初期メトリクスを確認する

**専門性の要件**: Kubernetes, CI/CD, デプロイ自動化

**期待成果物**: デプロイ実行ログ

**完了基準**: 全レプリカが正常に起動し、デプロイが完了している

### Step 5: ヘルスチェック

**実行内容**:
- アプリケーションのヘルスチェックエンドポイントを確認する:
  - `/health` または `/healthz`
  - `/ready`（Readiness Probe）
  - `/live`（Liveness Probe）
- 主要APIエンドポイントのスモークテストを実行する
- エラー率・レイテンシ・スループットを確認する:
  - エラー率: デプロイ前から有意な増加がないことを確認する
  - P95レイテンシ: SLO以内であることを確認する
  - スループット: 期待される処理量をさばけていることを確認する
- データベース接続・クエリ実行の正常性を確認する
- ログに異常なエラーが出力されていないことを確認する
- 外部依存サービスとの接続を確認する

**専門性の要件**: SRE、オブザーバビリティ、ヘルスチェック設計

**期待成果物**: ヘルスチェック結果（全項目OK/NG）

**完了基準**: 全ヘルスチェック項目がOKであり、エラー率・レイテンシがSLO以内である

### Step 6: 異常時ロールバック

**実行内容**:
（Step 5でNGが発生した場合に実施）

- ロールバック判断基準を確認する（エラー率・レイテンシ閾値の超過）
- アプリケーションをロールバックする:
  ```bash
  # Kubernetes
  kubectl rollout undo deployment/app
  # または特定リビジョンへ
  kubectl rollout undo deployment/app --to-revision=N
  ```
- DBマイグレーションのロールバックが必要かを判断する
  - DOWNマイグレーションを実行する（設計書の手順に従う）
- ロールバック後にヘルスチェックを再実施する
- インシデントレポートを作成する（原因・対応内容・再発防止策）

**専門性の要件**: インシデント対応、ロールバック手順

**期待成果物**: ロールバック実行ログとインシデントレポート

**完了基準**: ロールバックが完了し、システムがデプロイ前の状態に戻っている

### Step 7: リリースノート生成

**実行内容**:
- デプロイしたバージョン・コミット範囲を確認する
- 変更内容をカテゴリ別に整理する:
  - 新機能（Features）
  - バグ修正（Bug Fixes）
  - パフォーマンス改善（Performance）
  - セキュリティ修正（Security）
  - 非推奨（Deprecations）
  - 破壊的変更（Breaking Changes）
- ステークホルダー向けのリリースアナウンスを作成する
- デプロイ記録（日時・バージョン・実施者・結果）を記録する

**専門性の要件**: 技術文書作成、リリース管理

**期待成果物**: `docs/release-notes-{version}.md`

**完了基準**: リリースノートが生成され、デプロイ記録が残されている

## 品質基準

このSkillの完了条件:
- [ ] デプロイ前チェック全項目がOKになっている
- [ ] DBマイグレーションが正常完了し、データ整合性が確認されている
- [ ] アプリケーションデプロイが完了している
- [ ] 全ヘルスチェック項目がOKになっている
- [ ] エラー率・レイテンシがSLO以内である
- [ ] リリースノートが生成されている

## 成果物

- **デプロイ結果レポート**: インラインで報告
  - デプロイ実施日時・バージョン
  - 各ステップの実行結果
  - ヘルスチェック結果
  - 実施後のシステム状態
- **リリースノート**: `docs/release-notes-{version}.md`
  - 変更内容（機能/バグ修正/セキュリティ）
  - 影響範囲
  - ロールバック手順（緊急時用）

## デプロイ前チェックリスト

```
[ ] CIパイプライン: 全テストパス
[ ] セキュリティスキャン: Critical/High脆弱性なし
[ ] 統合テスト: Go判定
[ ] DBバックアップ: 最新バックアップ確認
[ ] マイグレーション設計書: レビュー済み（マイグレーションがない場合はN/A）
[ ] ロールバック手順: 確認済み
[ ] 関係者通知: 完了
[ ] オンコール担当: 待機中
```
