## 目的

要件定義書をインプットとして、実装可能な技術アーキテクチャを設計する。
テックスタック選定・DBスキーマ・OpenAPI仕様・アーキテクチャ決定記録（ADR）を生成し、
開発チームが即座に実装を開始できる状態にする。

## 実行プロセス

### Step 1: 要件レビューとアーキテクチャ制約の把握

**実行内容**:
- `docs/requirements.md`（または提供された要件）を読み込み、機能要件・非機能要件を把握する
- 既存システムの技術スタック・制約条件を確認する
- チームの技術スキルセット・学習曲線を考慮する
- 予算・タイムライン・運用体制の制約を確認する
- クラウドプロバイダー・インフラ環境の制約を確認する
- データプライバシー・コンプライアンス要件（GDPR, HIPAA等）を確認する

**専門性の要件**: システムアーキテクチャ、非機能要件分析

**期待成果物**: アーキテクチャ設計の制約と前提条件の整理

**完了基準**: 設計判断に影響する全ての制約が識別されている

### Step 2: テックスタック評価・選定

**実行内容**:
- 要件と制約に基づき、候補となるテックスタックを2〜3案提示する
- 各候補を以下の観点で評価する:
  - 要件適合性（機能要件・非機能要件への対応力）
  - 開発生産性（DX、エコシステム、学習コスト）
  - 運用性（監視、デプロイ、スケーリング容易性）
  - コスト（ライセンス、インフラ、開発コスト）
  - 技術的成熟度・コミュニティサポート
  - セキュリティ実績
- 推奨スタックを選定し、選定理由を明記する
- 採用しない候補についてもトレードオフを説明する

**専門性の要件**: テクノロジー評価、システムアーキテクチャ

**期待成果物**: テックスタック比較表と選定結果

**完了基準**: 推奨スタックが選定され、トレードオフが文書化されている

### Step 3: アーキテクチャパターン設計

**実行内容**:
- システムアーキテクチャパターンを選定する
  - モノリシック / モジュラーモノリス / マイクロサービス / サーバーレス / イベント駆動
- レイヤー構造を定義する
  - Presentation / Application / Domain / Infrastructure
  - Clean Architecture / Hexagonal Architecture / MVC等
- コンポーネント間の通信方式を設計する
  - 同期（REST/gRPC）/ 非同期（メッセージキュー）
  - イベントソーシング / CQRS の採否
- スケーリング戦略を設計する
  - 垂直スケーリング / 水平スケーリング
  - キャッシュ戦略（Redis, CDN等）
  - ロードバランシング戦略
- セキュリティアーキテクチャを設計する
  - 認証（JWT, OAuth2, OIDC等）
  - 認可（RBAC, ABAC等）
  - ネットワークセキュリティ

**専門性の要件**: ソフトウェアアーキテクチャ、分散システム設計

**期待成果物**: `docs/architecture.md`（コンポーネント図・デプロイ図・スケーリング戦略を含む）

**完了基準**: 全コンポーネントの役割と通信方式が定義されている

### Step 4: データベーススキーマ設計

**実行内容**:
- エンティティ定義に基づいてDBスキーマを設計する
- テーブル定義（カラム名・型・制約・インデックス）を作成する
- 正規化レベルを決定する（3NF/BCNF または非正規化の判断）
- リレーションシップとFK制約を定義する
- パーティショニング・シャーディング戦略を評価する
- インデックス戦略を設計する（クエリパターンに基づく）
- マルチテナント設計が必要な場合はテナント分離方式を定義する
- NoSQL使用の場合はドキュメント構造・コレクション設計を行う

**専門性の要件**: データベース設計、SQLパフォーマンス最適化

**期待成果物**: `docs/schema.md`（DDL含むDBスキーマ定義）

**完了基準**: 全テーブルのDDLとインデックス設計が完成している

### Step 5: OpenAPI仕様生成

**実行内容**:
- ユースケースに基づいてRESTful APIエンドポイントを設計する
  - リソース指向URL設計（/resources/{id}/sub-resources）
  - HTTPメソッドの適切な使用（GET/POST/PUT/PATCH/DELETE）
  - HTTPステータスコードの一貫した使用
- OpenAPI 3.0 形式で仕様を記述する:
  - 全エンドポイントのパスとメソッド
  - リクエストボディ・パラメータのスキーマ
  - レスポンスのスキーマ（200/201/400/401/403/404/500等）
  - 認証スキーム（Bearer JWT等）
  - ページネーション方式（cursor/offset）
- APIバージョニング戦略を定義する（URLパス / ヘッダー）
- レートリミット方針を定義する

**専門性の要件**: REST API設計、OpenAPI仕様

**期待成果物**: OpenAPI 3.0 仕様ファイル（下記パスルールに従う）
- プロジェクト内に既存の `openapi.yaml` / `openapi.json` がある場合 → そのファイルを更新する
- 見つからない場合 → `docs/openapi.yaml` に新規生成する

**完了基準**: 全ユースケースに対応するAPIエンドポイントがOpenAPI形式で定義されている

### Step 6: ADR（アーキテクチャ決定記録）作成

**実行内容**:
- 重要なアーキテクチャ決定を個別のADRとして文書化する
- 各ADRは以下を含む:
  - 状態（Proposed / Accepted / Deprecated / Superseded）
  - コンテキスト（なぜこの決定が必要だったか）
  - 決定（何を選択したか）
  - 結果（この決定がもたらすもの：利点・欠点・トレードオフ）
  - 代替案（検討した他の選択肢）
- 最低限ADRが必要な決定:
  - テックスタック選定
  - アーキテクチャパターン
  - 認証・認可方式
  - データベース選択
  - API設計原則

**専門性の要件**: アーキテクチャ文書作成、意思決定記録

**期待成果物**: `docs/adr.md`（全ADRを含む）

**完了基準**: 全ての重要アーキテクチャ決定がADRとして文書化されている

### Step 7: 実装タスク分割

**実行内容**:
- アーキテクチャ設計から実装タスクを分割する
- バックエンド・フロントエンド・インフラのタスクを並行化できるように整理する
- タスク間の依存関係を定義する
- 各タスクの実装に必要な技術スキルを明記する
- MVP（最小実行可能製品）に必要なタスクと後続フェーズのタスクを分類する

**専門性の要件**: プロジェクト管理、アジャイル開発

**期待成果物**: 実装タスク一覧（依存関係・担当スキル・MVP/Phase分類付き）

**完了基準**: 開発チームが即座に実装開始できるタスクリストが完成している

## 品質基準

このSkillの完了条件:
- [ ] テックスタックが選定され、トレードオフが文書化されている
- [ ] アーキテクチャパターンとコンポーネント構造が定義されている
- [ ] DBスキーマ（DDL含む）が設計されている
- [ ] OpenAPI 3.0 形式の API仕様が生成されている
- [ ] 重要アーキテクチャ決定がADRとして記録されている
- [ ] 実装タスクが分割・優先順位付けされている

## 成果物

- **ADR**: `docs/adr.md`
- **OpenAPI仕様**: 既存ファイルがあればそれを更新、なければ `docs/openapi.yaml` に新規生成
- **DBスキーマ**: `docs/schema.md`（DDL含む）
- **アーキテクチャ概要**: `docs/architecture.md`（コンポーネント図含む）
- **実装タスクリスト**: 開発チームへの引き渡し資料
