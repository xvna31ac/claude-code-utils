name: agent-review
description: |
  フォーマット・ガイドライン準拠・ロールタイプ整合性・意味的一貫性の観点でエージェント定義ファイル（.claude/agents/*.md）をレビューし、問題点を重大度別に検出します。
  エージェント定義の品質確認、新規作成エージェントのレビュー、既存エージェントの問題調査が必要な場合に自動適用されます。
  「エージェントをレビューして」「agent定義を確認して」「.claude/agentsの品質チェックをして」でトリガーされます。
---

## 概要

`.claude/agents/*.md` を対象に、エージェント定義ガイドラインと実レビュー経験から導出した観点で品質を検証する。
問題点を Critical（動作を壊す）/ Important（品質・運用性に影響）/ Nice to have（明確性向上）の3段階で分類して報告する。

**対象外**: `.claude/skills/*/SKILL.md`（スキルファイルはエージェントとは別物のため対象外）

---

### Step 1: 対象特定とタイプ自動識別

- 存在する場合: ユーザー指定のファイルまたはパスを使用する
- 存在しない場合: `.claude/agents/*.md` を Glob でスキャンして全ファイルを対象にする
- 各ファイルの tools フィールドを確認してロールタイプを以下のルールで識別する:
  - **コードライター型**: tools に Write/Edit/Bash のいずれかが含まれる
  - **リードオンリー型**: tools が Read/Grep/Glob のみ（Write/Edit/Bash/WebFetch/WebSearch なし）
  - **リサーチ型**: tools に WebFetch/WebSearch があり Write/Edit/Bash がない
  - **ドキュメント型**: tools に Write/Edit があり WebFetch/WebSearch も含まれる
  - **中間型**: 上記のいずれにも分類できない組み合わせ
- 完了基準: レビュー対象ファイルのパス一覧とロールタイプが確定している

※ Step 1 は標準4項目形式の例外。「存在する場合 / しない場合」の分岐形式を使う。

---

### Step 2: ガイドライン確認

- 実行内容: レビュー基準となるガイドラインを確認する
- 専門性:
  - 以下の優先順位でガイドラインを取得する:
    1. **カスタムガイドライン（ユーザー指定）**: ユーザーがガイドラインファイルを指定している場合はそれを Read する
    2. **CLAUDE.md**: プロジェクトルートの `CLAUDE.md` の存在を Glob で確認し、存在すれば Read する
    3. **フォールバック**: CLAUDE.md が存在しない場合は `.claude/skills/agent-review/references/agent-guidelines.md` を Read する
  - プロジェクト固有の追加観点（Step 4〜6）は全ケースで常に適用する
- 期待成果物: レビュー基準（ガイドライン）の内容
- 完了基準: いずれかのガイドラインを取得済みである

---

### Step 3: ガイドライン一般観点レビュー

- 実行内容: 対象ファイルを Read して、Step 2 で取得したガイドラインの全観点に基づいてレビューする
- 専門性: Step 2 で取得したガイドライン（CLAUDE.md / フォールバック / カスタム）を適用基準とする
- 期待成果物: ガイドライン違反の一覧（重大度付き）
- 完了基準: 全対象ファイルのガイドライン観点レビューが完了している

---

### Step 4: フォーマット検証

- 実行内容: Step 1 の識別結果に応じてフォーマット観点を切り替えて検証する
- 専門性:

  **必須フィールド検証:**
  - `name` / `description` / `tools` の3フィールドが存在するか
  - `name` がケバブケース（小文字英数字とハイフンのみ）か
  - `name` とファイル名（拡張子を除く）が一致しているか
  - `description` が quoted string（ダブルクォートまたはシングルクォート）か（`|` ブロックスカラーは不可）
  - `tools` の値がカンマ区切りで有効なツール名のみを含むか（有効値: Read, Write, Edit, Bash, Glob, Grep, WebFetch, WebSearch, Agent）
  - `tools` に重複がないか
  - `model` が指定される場合、sonnet/opus/haiku 系のいずれかか
  - **model とタスク複雑度の整合性**（Nice to have）:
    - 探索・検索・単純分析タスクに `opus` 系が指定されていないか（`haiku` 系が適切）
    - 複雑な設計・アーキテクチャ判断タスクに `haiku` 系が指定されていないか（`opus` 系が適切）
    - 不整合がある場合は Nice to have として報告する
  - `permissionMode` が指定される場合、有効値（`dontAsk` / `acceptEdits` / `bypassPermissions`）か（Important）
  - **permissionMode とロールタイプの整合性**（Important）:
    - リードオンリー型 → `dontAsk` が適切（Write 系ツールがないため許可確認不要）
    - コードライター型 → `acceptEdits` が基本（ファイル編集を自動承認）
    - `bypassPermissions` は全許可のため、意図的設定かどうかを Important として報告する
  - `memory` が指定される場合、有効値（`user` / `project`）か（Nice to have）

  **ロールタイプ別 tools 整合性:**
  - description の動詞・ニュアンスと tools を対照する:
    - 「レビュー」「確認」「監査」「分析」→ リードオンリー型が適切（Write/Edit/Bash は原則不要）
    - 「実装」「構築」「作成」「修正」→ コードライター型が適切（Write/Edit/Bash が必要）
    - 「調査」「リサーチ」「検索」→ リサーチ型が適切（WebFetch/WebSearch があり Write なし）
    - 「ドキュメント」「文書化」→ ドキュメント型が適切（Write/Edit と WebFetch/WebSearch を含む）
  - ロールタイプと tools の組み合わせが不整合な場合は Important として報告する

- 期待成果物: フォーマット違反の一覧（重大度付き）
- 完了基準: 全対象ファイルのフォーマット検証が完了している

---

### Step 5: 参照パス・運用正確性検証

- 実行内容: ファイル内の参照パスと運用時動作に影響する記述を検証する
- 専門性（参照パス）:
  - 参照パスがプロジェクトルート相対の完全パスか（例: `agents/foo.md` は不可 → `.claude/agents/foo.md` が正しい）
  - 「〜エージェントに連携」「〜エージェントを呼び出す」などの記述で参照されているエージェント名が `.claude/agents/` に実際に存在するか（Glob で確認）
  - `*.md` 等のワイルドカードが Read 参照に使われていないか（Read ツールはワイルドカード非対応）
- 専門性（運用正確性）:
  - body の冒頭ロール宣言（「あなたは〜です」）が description と矛盾していないか
  - 「コミュニケーションプロトコル」セクションの JSON ペイロードに `requesting_agent` フィールドが含まれる場合、その値が `name` フィールドと一致するか
- 期待成果物: 参照パス違反・運用上の問題点の一覧
- 完了基準: 全参照パスと運用正確性項目の検証が完了している

---

### Step 6: 意味的整合性・クロスファイル検証

- 実行内容: 記述の意味的一貫性と複数ファイル間の整合性を検証する
- 専門性（意味的整合性）:
  - **description の使用条件明確性**: 「いつこのエージェントを使うべきか」が description から明確に読み取れるか
  - **description の具体性（強い description）**（Important）: 役割名のみ（「Code reviewer」「Backend developer」など）にとどまらず、具体的なユースケース（「〜の場合に使用」「Use when...」等）を含むか。含まない場合は Important として報告する
  - **description と body 冒頭ロール宣言の一貫性**: description で宣言した役割と body の役割説明が矛盾していないか
  - **tools とロール記述の整合性**: body 内で「〜を実装する」と書かれているのに tools に Write がない、などの不整合がないか
  - **推奨システムプロンプト構造への準拠**（Nice to have）: body が「Role → When Invoked → Key Practices → Important Notes → Example Usages」の順で構成されているか。現状の「ロール宣言 / チェックリスト / Communication Protocol / Development Workflow」構造も許容するが、推奨構造に近い方が好ましい
  - **過度な詳細指定の回避**（Nice to have）: Claude が自明に知っている情報（言語仕様・フレームワークのデフォルト動作・一般的なベストプラクティス）を body 内で逐一指示していないか。過度に詳細な指定は柔軟性低下・メンテナンス負担増につながるため Nice to have として報告する
  - **Communication Protocol セクションの有無**: 他エージェントとの連携を前提とする場合、Communication Protocol セクションが存在するか（Nice to have）
  - **Development Workflow セクションの有無**: 複雑なタスクを実行するエージェントの場合、Development Workflow セクションが存在するか（Nice to have）
- 専門性（クロスファイル）:
  - 「他のエージェントとの連携」セクションで名前が挙げられているエージェントが `.claude/agents/` に実際に存在するか（Glob で確認）
  - 複数のエージェント定義に矛盾する役割定義がないか（同じ役割が重複して定義されていないか）
  - **CLAUDE.md との内容重複**（Nice to have）: Step 2 で取得した CLAUDE.md の内容と比較して、プロジェクト共通ルールがエージェント定義内に重複して記述されていないか。重複がある場合は Nice to have として報告する
- 期待成果物: 意味的整合性・クロスファイル整合性の問題点の一覧
- 完了基準: 全意味的整合性・クロスファイル項目の検証が完了している

---

### Step 7: AI 自己フィードバックループ（省略不可）

- 実行内容: Step 3〜6 の検証結果を自己評価して、レポート前に誤検知・見落としを排除する（※ このステップは省略不可。レポート前に必ず実施する）
- 専門性:
  1. **False Positive 排除**: 各指摘がファイル内に実際に存在するかを再確認する（「〜であるべきだが存在しない」という指摘は実際にファイルを再 Read して確認する）
  2. **重大度分類の妥当性**: Critical / Important / Nice to have の分類が適切かを再評価する（「動作を壊す可能性があるか」を基準に Critical を厳選する）
  3. **修正案の具体性**: 各指摘の修正案が「具体的・実行可能」かを確認する（「〜を検討する」は不可、「〜を〜に変更する」が必要）
  4. **全観点の再走査**: Step 3〜6 の全観点を走査して重大な見落としがないか確認する
- 期待成果物: 精査済みの指摘リスト（誤検知を除去し、修正案が具体化された状態）
- 完了基準: 全指摘が精査済みで、False Positive が除去されている

---

### Step 8: レポート提示と修正確認

- 実行内容: 精査済みの検証結果をチャット上でレポートとして提示し、修正を実施するか確認する
- 専門性: 問題点を重大度・検証カテゴリ別に整理し、修正の優先順位と具体的な修正案を明示する
- レポート形式:
  ```
  # エージェントレビューレポート

  ## サマリー
  - レビュー対象: {ファイルパス一覧}
  - ガイドライン: {使用したガイドライン（CLAUDE.md / フォールバック / カスタム）}
  - Critical: N件 / Important: N件 / Nice to have: N件
  - 総合評価: [要修正 / 軽微な修正推奨 / 問題なし]

  ## Critical（動作を壊す可能性）
  ### [ファイル名] 問題の概要
  - 問題: {具体的な問題の説明}
  - 該当箇所: {行番号または引用}
  - 修正案: {具体的な修正内容}

  ## Important（品質・運用性に影響）
  ...

  ## Nice to have（明確性向上）
  ...

  ## 問題なし
  - {問題が検出されなかった検証カテゴリ}
  ```
- レポート提示後に「修正を実施しますか？」とユーザーに確認する
- 期待成果物: チャット上のレビューレポートとユーザーへの修正確認
- 完了基準: レポートが提示され、ユーザーの意向（修正する / しない）が確認されている

---

### Step 9: 修正実施（ユーザーが要求した場合のみ）

- 実行内容: ユーザーが修正を承認した場合に、指摘事項を1件ずつ修正する
- 専門性:
  - Critical → Important → Nice to have の順で修正する
  - 各修正を Edit ツールで実施する
  - 各修正後に AI 自己フィードバックループを実施する:
    1. 修正が問題を実際に解決しているか（修正後ファイルを再 Read して確認）
    2. 修正が新たな問題を導入していないか（周辺の記述との整合性を確認）
    3. 他箇所への意図しない影響がないか
- 期待成果物: 修正済みエージェントファイル
- 完了基準: 承認された全指摘の修正が完了し、各修正が自己フィードバックループで検証済みである
