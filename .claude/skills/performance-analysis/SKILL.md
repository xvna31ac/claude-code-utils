## 目的

コードの計算量・メモリ使用・データベースクエリ・フロントエンド描画を静的解析し、
パフォーマンスボトルネックを特定して具体的な改善提案を含むレポートを生成する。

## 実行プロセス

### Step 1: パフォーマンスプロファイリング計画

**実行内容**:
- 対象コードのデータフロー・処理フローを把握する
- ホットパス（頻繁に呼び出される処理）を特定する
- データ量・リクエスト頻度の想定を確認する
- 既存のパフォーマンス計測ツール・APM設定を確認する
- パフォーマンス要件（SLO/SLA）の確認を行う

**専門性の要件**: アプリケーションパフォーマンス分析、システム設計

**期待成果物**: 分析対象のホットパス一覧とパフォーマンス要件の整理

**完了基準**: 分析すべき優先領域が特定されている

### Step 2: アルゴリズム計算量分析

**実行内容**:
- 各関数・メソッドの時間計算量（Big-O記法）を分析する
  - O(n²)以上のアルゴリズムをループ構造から特定する
  - 不必要なソートや全件スキャンを検出する
  - 再帰処理の深さとメモ化の必要性を評価する
- 空間計算量（メモリ使用）を分析する
  - 大規模データの不必要なコピーを検出する
  - バッファリングが必要な箇所を特定する
  - メモリリークの可能性を確認する
- 定数倍の最適化機会を特定する
  - 早期脱出（early return）が可能な条件分岐を確認する
  - ループ内の不変計算をループ外に移動できるかを確認する

**専門性の要件**: アルゴリズム・データ構造、計算量理論

**期待成果物**: 計算量問題一覧（Big-O分析付き）

**完了基準**: O(n²)以上の全アルゴリズムが特定され、改善案が提示されている

### Step 3: N+1クエリ検出

**実行内容**:
- ループ内でのデータベースクエリ実行を検出する（N+1問題）
- ORM使用箇所（ActiveRecord, Hibernate, SQLAlchemy等）でのLazy Loading問題を特定する
- Eager Loading（preload, includes, joinedload等）で解決可能な箇所を提案する
- クエリのインデックス利用状況を評価する（WHERE句、JOIN条件のインデックス有無）
- SELECT *の使用（不必要なカラム取得）を検出する
- 同一データの重複クエリをキャッシュで解決可能か評価する
- バルク操作（bulk_create, bulk_update等）で置き換えられる複数INSERT/UPDATEを検出する

**専門性の要件**: データベース最適化、ORM、SQLクエリ最適化

**期待成果物**: N+1問題・クエリ最適化機会の一覧（推定改善効果付き）

**完了基準**: 全てのN+1問題が特定され、Eager Loading実装例が提示されている

### Step 4: メモリ使用パターン評価

**実行内容**:
- 大規模データの一括ロード（メモリ上への全件展開）を検出する
- ストリーミング・ページネーション処理が適切に実装されているかを確認する
- クロージャによるメモリリークの可能性を評価する
- キャッシュの有効期限・メモリ上限設定を確認する
- オブジェクトプール・再利用パターンの適用機会を評価する
- ガベージコレクション負荷を高める処理パターンを特定する

**専門性の要件**: メモリ管理、ランタイム最適化

**期待成果物**: メモリ使用問題一覧と改善提案

**完了基準**: 潜在的なメモリリークと大規模メモリ使用箇所が特定されている

### Step 5: フロントエンド描画最適化（フロントエンドコードが対象の場合）

**実行内容**:
- 不必要な再描画（unnecessary re-renders）を引き起こすコードを特定する
  - React: useMemo/useCallback/React.memo の欠如
  - Vue: computed プロパティの未活用
  - Angular: ChangeDetectionStrategy.OnPush の未適用
- バンドルサイズ問題を評価する
  - 不必要なライブラリ全体のimport（tree-shaking非対応の使い方）
  - Code Splitting が適用されていない大規模コンポーネント
- 画像・アセットの最適化状況を確認する
- Critical Rendering Path のブロッキングを確認する
- 非同期処理（lazy loading）の適用機会を評価する

**専門性の要件**: フロントエンドパフォーマンス最適化、ブラウザレンダリング

**期待成果物**: フロントエンドパフォーマンス問題一覧（Core Web Vitals影響度付き）

**完了基準**: 不必要な再描画とバンドルサイズ問題が全て特定されている

### Step 6: キャッシュ戦略評価

**実行内容**:
- 現在のキャッシュ戦略（メモリ/Redis/CDN/HTTPキャッシュ）を評価する
- キャッシュ可能だがキャッシュされていない処理を特定する
- キャッシュの無効化戦略が適切かを評価する
- キャッシュスタンピード（thundering herd）問題のリスクを評価する
- APIレスポンスキャッシュ・マテリアライズドビューの適用機会を評価する

**専門性の要件**: キャッシュ設計、分散システム

**期待成果物**: キャッシュ最適化機会一覧

**完了基準**: キャッシュ未適用の高コスト処理が全て特定されている

### Step 7: 改善提案レポート生成

**実行内容**:
- 発見した問題を推定改善効果（High/Medium/Low）で分類する
- 各問題に対して具体的な改善コードを提示する
- 実装コスト対改善効果のマトリクスを作成する
- 優先度の高い改善から順に推奨実装順序を提示する

**専門性の要件**: パフォーマンスエンジニアリング、技術文書作成

**期待成果物**: パフォーマンス分析レポート

**完了基準**: 全ての問題に改善効果の推定値と具体的な改善コードが付与されている

## 品質基準

このSkillの完了条件:
- [ ] 計算量分析（O(n²)以上のアルゴリズム特定）が完了している
- [ ] N+1クエリ問題が全て特定されている
- [ ] メモリ使用パターンの問題が特定されている
- [ ] フロントエンドコードがある場合、描画最適化問題が評価されている
- [ ] 全ての問題に推定改善効果が付与されている
- [ ] 改善コードサンプルが具体的に提示されている

## 改善効果分類

| 効果 | 定義 |
|------|------|
| High | 応答時間50%以上短縮、またはスループット2倍以上向上 |
| Medium | 応答時間20-50%短縮、またはリソース使用量30%以上削減 |
| Low | 応答時間20%未満短縮、コードの将来的なスケーラビリティ改善 |
