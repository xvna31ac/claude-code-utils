## 目的

データベーススキーマ変更を、データ整合性を維持しながらゼロダウンタイム（または最小ダウンタイム）で
実施するためのマイグレーション計画を設計する。
UP/DOWNマイグレーションとロールバック手順を含む完全なマイグレーション設計書を生成する。

## 実行プロセス

### Step 1: 現在のスキーマ確認

**実行内容**:
- 現在のデータベーススキーマを確認する（テーブル定義・インデックス・制約・FK）
- 既存のマイグレーションファイルを確認し、現在のバージョンを特定する
  （ファイルが存在しない場合は新規プロジェクトとして扱い、バージョン 0 から開始する）
- 各テーブルの概算データ量を確認する（行数・テーブルサイズ）
- 使用しているORM・マイグレーションフレームワークを確認する
  - Rails: ActiveRecord Migrations
  - Python: Alembic, Django Migrations
  - Node.js: Knex, Flyway, Liquibase
  - Java: Flyway, Liquibase

**専門性の要件**: データベース管理、マイグレーションフレームワーク

**期待成果物**: 現在のスキーマ定義とマイグレーション履歴の整理

**完了基準**: 変更前の現状スキーマが完全に把握されている

### Step 2: スキーマ差分分析

**実行内容**:
- 目標スキーマ（変更後）と現在のスキーマの差分を分析する
- 変更タイプを分類する:
  - **追加**: 新規テーブル、カラム追加、インデックス追加、制約追加
  - **変更**: カラム型変更、カラム名変更、制約変更
  - **削除**: テーブル削除、カラム削除、インデックス削除
- 各変更の実施順序の依存関係を確認する（FK制約の追加順序等）
- DDL変更がどのロックレベルを必要とするかを評価する
  - PostgreSQL: AccessExclusiveLock vs ShareUpdateExclusiveLock
  - MySQL: メタデータロック、Online DDL対応可否

**専門性の要件**: データベース設計、DDL操作のロック動作

**期待成果物**: スキーマ差分一覧（変更タイプ・影響テーブル・ロックレベル付き）

**完了基準**: 全ての変更が分類され、DDLロックレベルが評価されている

### Step 3: データ影響評価

**実行内容**:
- 既存データへの影響を評価する
  - NOT NULL制約追加: デフォルト値または既存データ更新が必要
  - カラム型変更: データ変換の必要性と変換損失リスク
  - カラム削除: 削除対象カラムに存在するデータの扱い
  - テーブル分割・統合: データ移行スクリプトの必要性
- NULL値・デフォルト値の設定方針を決定する
- データ変換が必要な場合、変換ロジックの整合性を検証する
- 大規模データ更新が必要な場合はバッチ処理方式を設計する
  - バッチサイズの設定（デッドロック・パフォーマンス考慮）
  - 進捗トラッキング方式

**専門性の要件**: データ移行設計、データ整合性

**期待成果物**: データ影響評価結果とデータ変換方針

**完了基準**: 既存データへの全影響が評価され、変換方針が決定されている

### Step 4: UP/DOWNマイグレーション設計

**実行内容**:
- **UPマイグレーション**（現在→目標スキーマ）を設計する:
  - 実施順序を依存関係に基づいて決定する
  - 各ステップのSQL/DSLを記述する
  - ロールバックポイントを各ステップ後に設定する
- **DOWNマイグレーション**（目標→現在スキーマ）を設計する:
  - UPの逆順で実施できるように設計する
  - データ復元が必要な場合のバックアップ・リストア手順を含める
  - DOWNが実行不可能な変更（削除したデータの復元等）を明示する
- 各マイグレーションステップを以下の形式で記述する:
  ```sql
  -- UP: ステップ説明
  ALTER TABLE ...;
  -- DOWN: ステップ説明
  ALTER TABLE ...;
  ```

**専門性の要件**: DDL設計、データベース管理

**期待成果物**: UP/DOWN マイグレーションSQLの完全な実装

**完了基準**: 全ての変更がUP/DOWNで実装され、各ステップがアトミックになっている

### Step 5: ゼロダウンタイム手順策定

**実行内容**:
- **Expand-Contract パターン**を適用する（必要な場合）:
  1. **Expand（拡張）**: 新旧両方の構造をサポートするようにスキーマを拡張する
  2. **Migrate**: アプリケーションコードを新構造を使うように移行する
  3. **Contract（縮小）**: 古い構造を削除する
- 各フェーズでのアプリケーションコードの変更要件を定義する
- ブルーグリーンデプロイ・ローリングデプロイとの整合性を確認する
- マイグレーション実行中のパフォーマンス影響を評価する:
  - テーブルロック時間の推定
  - レプリケーション遅延リスク
  - アプリケーション応答時間への影響
- メンテナンスウィンドウが必要な変更を明示する

**専門性の要件**: ゼロダウンタイムデプロイ、分散システム

**期待成果物**: ゼロダウンタイム実施手順書

**完了基準**: 各変更のダウンタイム影響が評価され、ゼロダウンタイム実施手順が定義されている

### Step 6: ロールバック手順作成

**実行内容**:
- マイグレーション失敗時のロールバック手順を詳細に記述する
  - 各フェーズでのロールバック方法
  - アプリケーションコードのロールバックとDBロールバックの順序
  - データ復元が必要な場合のバックアップ・リストア手順
- ロールバック判断基準を定義する:
  - エラー率・レイテンシの閾値
  - ロールバック決定権者と連絡体制
  - ロールバック実施タイムリミット
- ロールバック不可能なポイントを明示する（Point of No Return）
- 事前バックアップのタイミングと方法を指定する

**専門性の要件**: 障害対応、リスク管理

**期待成果物**: ロールバック手順書（判断基準・連絡体制付き）

**完了基準**: 全フェーズでのロールバック手順が定義され、ロールバック不可点が明示されている

### Step 7: マイグレーション設計書生成

**実行内容**:
- 全ステップの成果物を統合した設計書を生成する
- 実施チェックリスト（本番実施前・実施中・実施後）を作成する
- 監視すべきメトリクス（レプリケーション遅延、エラー率等）を定義する
- 実施後の動作確認手順を作成する

**専門性の要件**: 技術文書作成、データベース運用

**期待成果物**: `docs/migration-design.md`

**完了基準**: 実施チームが設計書のみで安全にマイグレーションを実施できる

## 品質基準

このSkillの完了条件:
- [ ] 全てのスキーマ差分が分析されている
- [ ] データ影響評価が完了し、変換方針が決定されている
- [ ] UP/DOWNマイグレーションSQLが完全に記述されている
- [ ] ゼロダウンタイム実施手順が定義されている
- [ ] ロールバック手順が全フェーズで定義されている
- [ ] 実施チェックリストが作成されている
