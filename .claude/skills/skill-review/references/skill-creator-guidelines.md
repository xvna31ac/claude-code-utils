# skill-creator ガイドライン（フォールバック参照）

このファイルは skill-creator スキルが利用不可の場合のフォールバック参照として使用する。
skill-creator の核心原則を要約したものであり、レビュー観点の基準として機能する。

---

## 簡潔性優先（Concise is Key）

- Claude が既知の情報（言語仕様、フレームワークのデフォルト動作、一般的なベストプラクティス）を SKILL.md で繰り返さない
- 指示は Claude が「なぜそうするか」を知っている場合は省略し、「何をするか」だけを記述する
- 同じ情報を SKILL.md と references/ の両方に書かない（Progressive Disclosure 原則）

## 適切な自由度

- 脆弱性が高い箇所（フォーマット、必須フィールド）: 厳密に指定する
- 変動性が高い箇所（実装内容、スタイル選択）: Claude の判断に委ねる
- 指示の粒度は「Claude が間違えやすい箇所」に集中させる

## スキルの構造

```
.claude/skills/<name>/
  SKILL.md          # メイン定義（500行以内）
  references/       # 必要時のみロードする参照ファイル群
  scripts/          # 実行スクリプト（必要時のみ）
  assets/           # 静的アセット（必要時のみ）
```

**frontmatter（SKILL.md）:**
- `name`: スキル識別子（ケバブケース）
- `description`: 3文構成のブロックスカラー（`|` を使用）
- `tools` フィールドは禁止（スキルは Claude Code が自動的にツールを選択する）

## Progressive Disclosure

- SKILL.md は 500 行以内に収める
- 詳細なテンプレート・例・仕様は references/ に分離する
- references/ 内ファイルは「必要な Step で明示的に Read する」と記述する
- 大量の参照ファイルを冒頭で一括 Read させる記述は避ける

## description の品質基準

description は3文構成とする:
1. **役割文**: 動詞（「〜します」「〜して」）で始まり、スキルが「何をするか」を具体的に述べる
2. **適用条件文**: 「〜が必要な場合に自動適用されます」の形式で「いつ使われるか」を述べる
3. **トリガー文**: 「「〜して」「〜をお願い」などのリクエストでトリガーされます」の形式で具体的なトリガー例を列挙する

**禁止事項:**
- body 内に「## 適用条件」「## 使用場面」「## When to Use」セクションを設ける（description で完結させる）
- description に `>` スカラーを使用する（改行がスペースに変換され3文構成が崩れる）

## body の記述スタイル

- Step の「実行内容」: 命令形または不定詞形（「〜する」「〜を確認する」）
- 「専門性」項目: 具体的な検証条件を箇条書きで列挙する
- 「期待成果物」: 生成されるファイルまたは情報の具体的な形式を示す
- 「完了基準」: Step 完了の判断基準を一文で示す

## 標準 Step 構成（個別スキル）

各 Step は以下の4項目を含む:
```
- 実行内容: {何をするか}
- 専門性: {どのような観点で・どのような基準で}
- 期待成果物: {何が生成されるか}
- 完了基準: {いつ完了とみなすか}
```

**例外**: Step 1（前提確認）で分岐形式（「存在する場合 / しない場合」）を使う場合は、
その Step に「※ 標準4項目形式の例外」と注記する。

## 禁止事項

- `README.md` や補助ドキュメントを作成する（SKILL.md に統合する）
- SKILL.md と references/ で同じ内容を重複して記述する
- 一度スキャンした結果を後フェーズで再スキャンさせる記述をする
- 参照パスに `*.md` 等のワイルドカードを使用する（Read ツール非対応）
- プロジェクトルート相対でない不完全なパスを参照に使用する

## ワークフロースキル固有の要件

- name が `workflow-` プレフィックスで始まること
- フェーズ一覧テーブル（`| Phase | フェーズ名 | 使用Skill / 担当ロール | ... |`）を持つこと
- 承認ゲートに「確認内容 / APPROVE 時 / REJECT 時」を明記すること
- 状態ファイルパスが `.workflow/state/<name>.md` 形式であること
- 個別スキル呼び出しを `skill-name` 形式で参照すること（`/skill-name` は誤り）
