name: skill-review
description: |
  フォーマット・ガイドライン準拠・参照パス・意味的整合性の観点でスキルファイル（SKILL.md）をレビューし、問題点を重大度別に検出します。
  スキル定義の品質確認、新規作成スキルのレビュー、既存スキルの問題調査が必要な場合に自動適用されます。
  「スキルをレビューして」「SKILL.md を確認して」「スキルの品質チェックをして」でトリガーされます。
---

## 概要

`.claude/skills/*/SKILL.md` を対象に、skill-creator ガイドラインと実レビュー経験から導出した観点で品質を検証する。
問題点を Critical（動作を壊す）/ Important（品質・運用性に影響）/ Nice to have（明確性向上）の3段階で分類して報告する。

**対象外**: `.claude/agents/*.md`（エージェントファイルはスキルとは別物のため対象外）

---

### Step 1: 対象特定とタイプ自動識別

- 存在する場合: ユーザー指定のファイルまたはパスを使用する
- 存在しない場合: `.claude/skills/*/SKILL.md` を Glob でスキャンして全ファイルを対象にする
- 各ファイルのタイプを以下のルールで識別する:
  - **個別スキル**: `skills/*/SKILL.md`（スキル名が `workflow-` で始まらない）
  - **ワークフロースキル**: `skills/workflow-*/SKILL.md`
- 完了基準: レビュー対象ファイルのパス一覧とタイプが確定している

※ Step 1 は標準4項目形式の例外。「存在する場合 / しない場合」の分岐形式を使う。

---

### Step 2: ガイドライン確認

- 実行内容: レビュー基準となるガイドラインを確認する
- 専門性:
  - 以下の優先順位でガイドラインを取得する:
    1. **カスタムガイドライン（ユーザー指定）**: ユーザーがガイドラインファイルを指定している場合はそれを Read する（skill-creator の代替として使用する）
    2. **skill-creator**: `.claude/skills/skill-creator/SKILL.md` の存在を Glob で確認し、存在すれば Read する
    3. **フォールバック**: skill-creator が存在しない場合は `.claude/skills/skill-review/references/skill-creator-guidelines.md` を Read する
  - プロジェクト固有の追加観点（Step 4〜6）は全ケースで常に適用する
- 期待成果物: レビュー基準（ガイドライン）の内容
- 完了基準: いずれかのガイドラインを取得済みである

---

### Step 3: skill-creator 一般観点レビュー

- 実行内容: 対象ファイルを Read して、Step 2 で取得したガイドラインの全観点に基づいてレビューする
- 専門性: Step 2 で取得したガイドライン（skill-creator / フォールバック / カスタム）を適用基準とする
- 期待成果物: ガイドライン違反の一覧（重大度付き）
- 完了基準: 全対象ファイルのガイドライン観点レビューが完了している

---

### Step 4: タイプ別フォーマット検証

- 実行内容: Step 1 の識別結果に応じてフォーマット観点を切り替えて検証する
- 専門性:

  **個別スキル（メイン対象）:**
  - frontmatter に `name` と `description` のみか（`tools` フィールドが含まれていないか）
  - description が `|` ブロックスカラーで書かれているか（`>` は改行をスペースに変換するため不可）
  - description が3文構成か（1文目: 動詞で始まる役割説明 / 2文目: 「〜が必要な場合に自動適用されます」 / 3文目: トリガー例）
  - body の各 Step が「実行内容 / 専門性 / 期待成果物 / 完了基準」の4項目を含むか
  - Step 1（前提確認）が分岐形式（存在する場合 / しない場合）を使っている場合、例外であることが注記されているか
  - body に「適用条件」「使用場面」「When to Use」セクションが書かれていないか（description で完結させるべき）

  **ワークフロースキル:**
  - name が `workflow-` プレフィックスで始まるか
  - フェーズ一覧テーブル（`| Phase | フェーズ名 | 使用Skill / 担当ロール | ... |`）が存在するか
  - 承認ゲートに「確認内容 / APPROVE 時 / REJECT 時」が明記されているか
  - 状態ファイルパスが `.workflow/state/<name>.md` 形式か
  - 個別スキル呼び出しが `skill-name` 形式か（`/skill-name` は誤り）

- 期待成果物: フォーマット違反の一覧（重大度付き）
- 完了基準: 全対象ファイルのタイプ別フォーマット検証が完了している

---

### Step 5: 参照パス・運用正確性検証

- 実行内容: ファイル内の参照パスと運用時動作に影響する記述を検証する
- 専門性（参照パス）:
  - 参照パスがプロジェクトルート相対の完全パスか（例: `references/foo.md` は不可 → `.claude/skills/xxx/references/foo.md` が正しい）
  - `*.md` のようなワイルドカードが Read 参照に使われていないか（Read ツールはワイルドカード非対応）
  - 参照先ファイルが実際に存在するか（Glob で確認）
  - 「参照ファイル」セクションの列挙と、各フェーズで実際に Read する記述が一致しているか
- 専門性（運用正確性）:
  - **冪等性チェックの配置**: ファイル上書きの確認が「各ファイル生成直前に都度実施」と明記されているか（「生成手順の第1ステップ」として記述されると一括チェック後全生成と誤解される）
  - **ファイル生成タイミング**: 「どのフェーズで・どの条件で生成するか」が明確か
  - **スキップ条件**: 前フェーズで必要情報が得られる場合に確認・ステップを省略してよいと明示されているか
  - **スキャン重複の回避**: 前フェーズで取得済みのスキャン結果を後フェーズで再利用する記述があるか
- 期待成果物: 参照パス違反・運用上の問題点の一覧
- 完了基準: 全参照パスと運用正確性項目の検証が完了している

---

### Step 6: 意味的整合性・クロスファイル検証

- 実行内容: 記述の意味的一貫性と、references/ 内・複数ファイル間の整合性を検証する
- 専門性（意味的整合性）:
  - **成果物セクションの分類**: 「スキル自体のパッケージファイル（事前作成済み）」と「スキル実行時に生成されるファイル」が同一リストに混在していないか
  - **リスト内の項目混在**: 箇条書きに性質が異なる項目が同列で並んでいないか
  - **Phase 番号の整合性**: "Phase 6.5" のような半端な番号が使われていないか
  - **承認ゲートの配置**: 破壊的変更前・設計確定の場面にゲートがあるか。読み取り・分析のみのフェーズに不要なゲートが設置されていないか
  - **条件分岐の完全性**: 条件付きフェーズ・条件付き生成に「条件を満たさない場合の動作」が明示されているか
- 専門性（クロスファイル — references/ が存在するスキルのみ）:
  - **テンプレートと生成例の一致**: references/ 内の frontmatter テンプレートと生成例の形式が一致しているか
  - **SKILL.md と references/ の規則整合**: SKILL.md のフォーマット規則と references/ のテンプレート・例が矛盾していないか
  - **ワークフロースキルのスキル名参照**: 参照しているスキル名が `.claude/skills/` に実際に存在するか
- 期待成果物: 意味的整合性・クロスファイル整合性の問題点の一覧
- 完了基準: 全意味的整合性・クロスファイル項目の検証が完了している

---

### Step 7: AI 自己フィードバックループ（省略不可）

- 実行内容: Step 3〜6 の検証結果を自己評価して、レポート前に誤検知・見落としを排除する（※ このステップは省略不可。レポート前に必ず実施する）
- 専門性:
  1. **False Positive 排除**: 各指摘がファイル内に実際に存在するかを再確認する（「〜であるべきだが存在しない」という指摘は実際にファイルを再 Read して確認する）
  2. **重大度分類の妥当性**: Critical / Important / Nice to have の分類が適切かを再評価する（「動作を壊す可能性があるか」を基準に Critical を厳選する）
  3. **修正案の具体性**: 各指摘の修正案が「具体的・実行可能」かを確認する（「〜を検討する」は不可、「〜を〜に変更する」が必要）
  4. **全観点の再走査**: Step 3〜6 の全観点を走査して重大な見落としがないか確認する
- 期待成果物: 精査済みの指摘リスト（誤検知を除去し、修正案が具体化された状態）
- 完了基準: 全指摘が精査済みで、False Positive が除去されている

---

### Step 8: レポート提示と修正確認

- 実行内容: 精査済みの検証結果をチャット上でレポートとして提示し、修正を実施するか確認する
- 専門性: 問題点を重大度・検証カテゴリ別に整理し、修正の優先順位と具体的な修正案を明示する
- レポート形式:
  ```
  # スキルレビューレポート

  ## サマリー
  - レビュー対象: {ファイルパス一覧}
  - ガイドライン: {使用したガイドライン（skill-creator / フォールバック / カスタム）}
  - Critical: N件 / Important: N件 / Nice to have: N件
  - 総合評価: [要修正 / 軽微な修正推奨 / 問題なし]

  ## Critical（動作を壊す可能性）
  ### [ファイル名] 問題の概要
  - 問題: {具体的な問題の説明}
  - 該当箇所: {行番号または引用}
  - 修正案: {具体的な修正内容}

  ## Important（品質・運用性に影響）
  ...

  ## Nice to have（明確性向上）
  ...

  ## 問題なし
  - {問題が検出されなかった検証カテゴリ}
  ```
- レポート提示後に「修正を実施しますか？」とユーザーに確認する
- 期待成果物: チャット上のレビューレポートとユーザーへの修正確認
- 完了基準: レポートが提示され、ユーザーの意向（修正する / しない）が確認されている

---

### Step 9: 修正実施（ユーザーが要求した場合のみ）

- 実行内容: ユーザーが修正を承認した場合に、指摘事項を1件ずつ修正する
- 専門性:
  - Critical → Important → Nice to have の順で修正する
  - 各修正を Edit ツールで実施する
  - 各修正後に AI 自己フィードバックループを実施する:
    1. 修正が問題を実際に解決しているか（修正後ファイルを再 Read して確認）
    2. 修正が新たな問題を導入していないか（周辺の記述との整合性を確認）
    3. 他箇所への意図しない影響がないか
- 期待成果物: 修正済みスキルファイル
- 完了基準: 承認された全指摘の修正が完了し、各修正が自己フィードバックループで検証済みである
