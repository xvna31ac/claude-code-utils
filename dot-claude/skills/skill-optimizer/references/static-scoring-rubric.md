# 静的品質スコアリング採点基準

skill-optimizerのStep 4で使用する。empirical（動的eval）が担う仕様精度・出力拘束力はここでは評価しない。

総合点 = 5軸の合計（100点満点）

---

## 軸1: 専門知識網羅性（30点）

対象ドメインの業界標準・ベストプラクティス・重要な手法をどれだけカバーしているかを評価する。

### スキルタイプによる「専門知識」の解釈

| スキルタイプ | 判定方法 | 「業界標準」の解釈 |
|------------|---------|----------------|
| ドメインスキル | 外部標準が存在する（security・accessibility・API設計等） | 外部標準（OWASP・WCAG等）との照合 |
| プロセス・メタスキル | 外部標準がない（skill評価・workflow設計・prompt tuning等） | 採用した手法・理論の根拠の明確さ、および先行研究・業界知見の参照度で評価 |

**プロセス・メタスキルの採点観点**: 手法の選択理由が明示されているか / 代替アプローチへの言及があるか / 理論的背景（empirical eval・RED-GREEN-REFACTOR等）が正しく参照されているか

---

### 更新頻度による事前確認（ドメインスキルのみ）

| 頻度 | ドメイン例 | 対応 |
|------|----------|------|
| 高（毎年更新） | security、AI/ML安全性、cloud、LLMフレームワーク | WebSearchで最新標準を確認 |
| 中（数年周期） | accessibility（WCAG）、API設計、モバイルOS | WebSearchで現行バージョンを確認 |
| 低（ほぼ安定） | SOLID原則、Git運用、コード品質、テスト手法 | 訓練知識で評価 |

**判断基準**: 標準の主要バージョンが2年以内に変わった可能性があればWebSearchを使用する。判断に迷う場合は使用する。

### 採点基準

| 点数 | 基準 |
|------|------|
| 25〜30点 | 業界標準の主要項目を90%以上カバー。最新バージョンに準拠している |
| 18〜24点 | 主要項目の70〜89%をカバー。一部の重要な観点が欠落している |
| 10〜17点 | 主要項目の50〜69%をカバー。複数の重要な観点が欠落している |
| 0〜9点 | 主要項目の50%未満。ドメイン固有の専門知識がほぼない |

**採点根拠の記載方法**: 「[標準名]の[N]項目中[M]項目をカバーしているため[X]点」のように具体的に記載する。

---

## 軸2: 柔軟性・汎用性（25点）

特定の例に過適合せず、多様なユーザー・コンテキスト・edge caseに対応できるかを評価する。

### 採点基準

| 点数 | 基準 |
|------|------|
| 21〜25点 | 特定例への言及が最小限。抽象度が適切で様々なユースケースに適用可能。edge caseの扱いが明示されている |
| 15〜20点 | 概ね汎用的だが、特定のツール・フレームワーク・プロジェクトに依存している箇所がある |
| 8〜14点 | 特定の例やプロジェクトを前提とした記述が多い。応用が難しい |
| 0〜7点 | 特定のユースケースにしか通用しない。汎用性がほぼない |

### 確認観点

- 固有名詞（プロジェクト名・ファイルパス・人名）への依存度
- 「〜の場合は〜する」という条件分岐の適切さ
- 想定外の入力に対するフォールバックの有無
- whyの説明（理由があると状況に応じて正しく判断できる）

---

## 軸3: 実用価値（20点）

このスキルがない場合と比較して、どれだけ業務を削減・自動化できるかを評価する。

### 採点基準

| 点数 | 基準 |
|------|------|
| 17〜20点 | Before/Afterが明確。スキルなしでは30分以上かかる作業を大幅に自動化している |
| 11〜16点 | 一定の業務削減効果がある。一部の手動作業が残る |
| 5〜10点 | 効果は限定的。スキルなしでも大きな差がない |
| 0〜4点 | 業務削減効果がほぼない、または既存の方法と変わらない |

### 確認観点

- 繰り返し発生するタスクを対象としているか
- 人間が判断・手動でやると時間がかかる部分を自動化しているか
- scripts/を活用して決定論的な処理を内包しているか

---

## 軸4: 保守性（15点）

変更が必要になった時の影響範囲が小さく、構造がDRYかどうかを評価する。

### 採点基準

| 点数 | 基準 |
|------|------|
| 13〜15点 | DRY原則を徹底。同じ情報がSKILL.mdとreferences/に重複していない。変更箇所が局所的 |
| 8〜12点 | 概ねDRYだが、一部の情報が重複している |
| 4〜7点 | 重複が多い。1つの変更が複数箇所に影響する |
| 0〜3点 | 同じ情報がいたるところに書かれている。保守コストが高い |

### 確認観点

- SKILL.mdとreferences/で同じ内容を繰り返していないか（Progressive Disclosure原則）
- ドメイン知識の変化（標準バージョンアップなど）に対して変更箇所が最小限か
- references/への分離が適切に行われているか
- **設計思想の一貫性**: 同じ目的を果たす要素がスキル全体で統一された設計原則で実装されているか（例: 一部は動的・一部は静的、一部は参照ファイル・一部はSKILL.md直書きなど哲学的に混在していないか）
- **philosophy.md との整合性**: philosophy.md が存在する場合、SKILL.md 本文・references/ の内容が philosophy.md の記述と矛盾していないか（philosophy.md が権威）。矛盾がある場合は大幅減点
- **トークン効率**: 頻繁にロードされるスキルほど密度（情報量/行数）が重要。よく使うスキルで冗長な説明・繰り返しが多い場合は減点。目安: 頻繁使用スキルは200行以内、その他は500行以内

---

## 軸5: 補助資材（10点）

references/・templates/・scripts/・philosophy.md の充実度と活用の適切さを評価する。

### 採点基準

| 点数 | 基準 |
|------|------|
| 9〜10点 | 必要な補助資材が揃っている。「必要なStepで明示的にReadする」形式で参照されている。300行超のファイルには目次がある |
| 6〜8点 | 補助資材はあるが、一部のStepで必要なものが欠けている、または参照方法が不適切 |
| 3〜5点 | 補助資材が最小限。スキルの性質上あった方が良い資材が存在しない |
| 0〜2点 | 補助資材がほぼない、またはSKILL.md冒頭で一括Readさせている（Progressive Disclosure違反） |

### 確認観点

- scripts/に繰り返し実行される決定論的処理が含まれているか
- templates/にサンプル・スキーマ・フォーマット定義が含まれているか
- 大量参照ファイル（300行超）に目次（TOC）があるか
- ワイルドカードを使った参照がないか（Read非対応）
- **philosophy.md の存在**: スキルルート直下に philosophy.md があればプラス評価（設計意図が明文化・確認済み）。なくても減点しないが、Part 0 で矛盾が多いにも関わらず philosophy.md がない場合は「設計意図が不明確なまま運用されている」として減点対象とする

| グレード | 点数範囲 |
|---------|---------|
| S | 90〜100点 |
| A | 80〜89点 |
| B | 70〜79点 |
| C | 60〜69点 |
| D | 59点以下 |

## 注意事項

動的evalとスコアが矛盾する場合（例: 柔軟性スコアが高いが裁量補完が多い）は、動的evalを優先してStep 5の改善計画に「⚠️ 動的eval優先」を明記する。スコアは参考値として扱う。
