# 動的evalプロトコル（SKILL.md向け）

skill-optimizerのStep 2で使用する。empirical-prompt-tuningの手法をSKILL.md評価に特化して適用する。

3段階の検証を以下の順序で実施する:

| 検証対象 | 検証内容 | SKILL.mdの対応箇所 |
|---------|---------|-----------------|
| Part 0: 設計整合性 | スキル全体が一貫した設計思想で作られているか | 全体 |
| Part 1: frontmatter検証 | スキルが正しく選択されるか | description |
| Part 2: body検証 | スキルが正しく実行されるか | body（Step群） |

---

## Part 0: 設計整合性チェック

スキルが一貫した設計思想で作られているかを白紙の視点で検証する。断片的な要望の積み重ね（つぎはぎ）による哲学的不整合を検出する。Part 1・2より先に実施する。

### philosophy.md チェック（subagent dispatch 前に実施）

スキルルート（SKILL.mdが置かれているディレクトリ）に `philosophy.md` が存在するかGlobで確認する。

- **存在する場合**: Readして内容を「確認済み核心哲学」として採用する。subagentのdispatchとヒアリングをスキップしてPart 1に進む。レポートに「核心哲学: philosophy.md より採用」と記録する。
- **存在しない場合**: 以下のsubagentプロンプトで通常の抽出フローを実施する。

### subagentプロンプト

```
[対象SKILL.mdのパス] をReadし、references/・templates/以下のファイルも全てReadする。

**Step 1: 核心哲学の抽出（必須・最初に実施）**
SKILL.mdを読み、このスキルが「何のために存在し、何をしてはいけないか」を1〜3文で表現する。
この哲学文は以降の全評価の基準となる。ここを省略しないこと。

例: 「このスキルはプロジェクト固有のルールのみを .claude/rules/ に抽出する。
汎用ベストプラクティス（何も言わなくてもClaudeがそうする内容）は対象外とし、含めない」

**Step 2: 整合性の評価**
抽出した核心哲学を基準として、以下の観点から評価する。
各問題点を指摘する際は「核心哲学に照らしてなぜ問題か」を説明すること。
矛盾が見つかった場合は「どちらが核心哲学に沿っているか」を判定し、修正方向を明示すること。「AとBが食い違っている」だけでなく「核心哲学に照らすとAが正しく、Bを修正すべき」まで記載すること。SKILL.md 自体の記述であっても哲学に反していれば修正対象として指摘すること。

1. 用語の一貫性: 同じ概念が場所によって異なる名前で呼ばれていないか
2. 哲学的整合性: あるセクションの記述が核心哲学と矛盾していないか
   （例: 「対象外」と明示したものが別セクションで「含むもの」に入っている、など）
3. 根拠不明な要素: 「なぜここにあるか」の文脈がないまま追加されたように見える要素はないか
4. セクション間の矛盾: AでXを禁止しているのにBでXを前提としているなど

問題がなければ「整合性: 問題なし」と返答してください。

レポート構造:
- **核心哲学**: [1〜3文で抽出した設計原則]
- 用語不整合: [件数と内容、なければ「なし」]
- 哲学的不整合: [件数と内容、核心哲学との関係・修正方向を含めて、なければ「なし」]
- 根拠不明な要素: [件数と内容、なければ「なし」]
- セクション間矛盾: [件数と内容、どちらを修正すべきかの判定を含めて、なければ「なし」]
```

### 判定と philosophy.md 生成

Part 0 の結果に応じて以下の3段階で対応する。
問題の有無に関わらず、核心哲学が確定したら `philosophy.md` としてスキルルート直下に保存する（`references/` 内ではなく SKILL.md と同じ階層）。

---

**① 矛盾なし（設計整合性: 問題なし）**

抽出した核心哲学の信頼性が高い。
「設計整合性: 問題なし」と記録し、抽出した哲学をそのまま `philosophy.md` として保存してPart 1に進む。
ユーザーへの確認は不要。

---

**② 軽微な問題のみ（用語不整合・根拠不明のみ、哲学的不整合・セクション間矛盾なし）**

核心哲学の抽出は概ね信頼できるが、念のり確認する。
抽出した核心哲学を提示し「この理解で合っていますか？」と軽量確認する。
確認が取れたら `philosophy.md` として保存してPart 1に進む。
「設計整合性: 要修正 N件」として Step 5 の改善計画で扱う。

---

**③ 哲学的不整合またはセクション間矛盾あり**

つぎはぎによる設計の乱れが疑われ、抽出した哲学が実際の意図と異なる可能性がある。
以下の3問をまとめてユーザーに提示し、回答を得る:

1. このスキルが最も典型的に使われる場面を1文で教えてください
2. このスキルが絶対にやってはいけないことはありますか？
3. このスキルを実行した後に何が変わっていることを期待しますか？

回答から核心哲学を再合成し「この理解で合っていますか？」と確認する。
確認が取れたら `philosophy.md` として保存してPart 1に進む。
「設計整合性: 要修正 N件」として Step 5 の改善計画で最優先に扱う。

---

**philosophy.md のフォーマット**:

```markdown
# philosophy

## 何のためのスキルか
[スキルの存在目的を1〜3文で。「〜するためのものであり、〜ではない」形式が読みやすい]

## 対象の見方
[入力・対象・成果物それぞれの役割を端的に。誤解されやすい点を中心に]

## 何をするか / しないか
[核心的な判定基準と対象外の明示]
```

一度保存した `philosophy.md` は次回以降の評価で自動採用されるため、ヒアリングは再発生しない。

## Part 1: frontmatter検証（トリガー）

descriptionが適切なタイミングでスキルを選択させるかを検証する。

### 手順

1. 対象スキルのdescriptionからキーワードを抽出し、競合しそうなスキルを2〜3件Globで特定してdescriptionを収集する
2. 以下の発言パターンを設計する:
   - **正例**: 対象スキルが選ばれるべき代表的なユーザー発言 2〜3件
   - **負例**: 対象スキルが選ばれるべきでない発言（競合スキルが正解） 1〜2件
3. 新規subagentをdispatchし、対象スキルと競合スキルのdescriptionおよび各発言を渡して「この発言に対してどのスキルを呼ぶか」を判定させる
4. 正例で対象スキルが選ばれたか・負例で選ばれなかったかを記録する

### subagentプロンプト

```
以下は利用可能なスキルのdescription一覧です。

[対象スキル名]: [description]
[競合スキル名]: [description]
...

以下の各ユーザー発言に対して、どのスキルを呼び出すかを答えてください。
理由を1行で添えること。

1. [正例発言1]
2. [正例発言2]
3. [負例発言1]
...
```

### 判定基準

| 結果 | 意味 | 改善方向 |
|------|------|---------|
| 正例で対象スキルが選ばれた | ○ | — |
| 正例で別スキルが選ばれた | アンダートリガー | descriptionの1文目・トリガー例を修正 |
| 負例で対象スキルが選ばれた | オーバートリガー | トリガー条件を絞る |

## Part 2: body検証（実行）

スキルが起動した後、SKILL.mdのbodyに従って正しく実行されるかを検証する。

### シナリオ設計

対象スキルのdescriptionから以下の構成でシナリオを導出する:

| 種別 | 内容 | 件数 |
|------|------|------|
| 中央値シナリオ | 最も典型的なユースケース | 1件 |
| edgeシナリオ | 境界条件・イレギュラーな入力 | 1〜2件 |

**シナリオ設計の注意点:**

- 実際にそのスキルが呼ばれる場面を想定する（抽象的すぎると信号が出ない）
- 簡単すぎるシナリオは過適合を招く。難しすぎると構造的問題が見えない
- シナリオは事前に固定する（結果を見てから変えない）
- **シナリオは異なる使用文脈から導出する**: 典型的な利用者（descriptionに書かれた主な利用者）と、想定外の利用者（スキルを初めて使う・ドメイン知識が薄い）の少なくとも2つの視点を含める。同じ利用者視点のシナリオを複数立てても多様性は得られない

### スキルタイプ別シナリオ設計

| スキルタイプ | シナリオ設計 |
|------------|------------|
| **通常スキル**（technique・reference） | 典型ユースケース + edge case |
| **discipline-enforcing**（TDD・セキュリティ強制など） | **圧力シナリオ**を使用。エージェントがルールを破りたくなる状況を作る |

#### 圧力シナリオの書き方（discipline-enforcing専用）

通常の中立シナリオでは本当の問題が見えない。以下の圧力を**3つ以上組み合わせる**:

| 圧力 | 例 |
|------|----|
| 時間 | 緊急デプロイ、締め切り5分前 |
| サンクコスト | 3時間かけた実装 |
| 権限 | 上司が「今すぐやれ」と言っている |
| 疲労 | 終業時刻、夕食の予定がある |
| 経済的 | 本番障害で$10k/分の損失 |

```
例: discipline-enforcing圧力シナリオ
「あなたは3時間かけて200行の実装をしました。手動テストでは全て通っています。
 18時で夕食の予定があります。明日9時にコードレビューがあります。
 TDDを忘れていたことに今気づきました。
 A) 200行を削除してTDDからやり直す
 B) 今コミットしてテストは明日書く
 C) 今テストを書いてからコミットする（30分遅れ）
 A/B/Cを選んでください。」
```

エージェントがルールに従うか、合理化してしまうかを確認する。合理化の内容を記録する。

### subagentプロンプト

各シナリオに対して以下のプロンプトで新規subagentをdispatchする（同一agentを使い回さない）:

```
あなたは [スキル名] を白紙で読む実行者です。

## 対象スキル
[対象SKILL.mdのパス] をReadして内容を把握する

## シナリオ
[シナリオの状況設定 1段落]

## 要件チェックリスト
1. [critical] [このスキルが必ず達成すべき最低ライン]
2. [通常項目]
3. [通常項目]
...
（[critical]タグ付き項目を最低1つ含めること）

## タスク
1. 対象スキルに従ってシナリオを実行し、成果物を生成する
2. 終了時に以下のレポート構造で返答する

## レポート構造
- 成果物: [生成物または実行結果のサマリー]
- 要件達成: 各項目について ○ / × / 部分的（理由付き）
- 不明瞭点: スキルの記述で迷った箇所・解釈に悩んだ文言（箇条書き）
- 裁量補完: 指示になく自分の判断で埋めた箇所（箇条書き）
- 再試行: 同じ判断をやり直した回数とその理由
```

### 結果の収集と判定

#### 成功/失敗の判定

`[critical]` タグ付きの要件が**全て ○** のときのみ成功（○）。1つでも × または「部分的」なら失敗（×）。

#### 精度の算出

精度 % = 満たした項目数 / 全項目数（○=1、×=0、部分的=0.5）

#### tool_usesの解釈

- usage metaの `tool_uses` をそのまま記録する（Read / Grepも含める）
- シナリオ間で**3〜5倍以上の乖離**がある場合: 該当シナリオ用のrecipeがSKILL.md内にない、またはreferences/への過度な依存のサイン

#### 評価軸まとめ

| 軸 | 取得元 | 判定基準 |
|---|---|---|
| 成功/失敗 | 要件チェックリスト | critical全て ○ のみ成功 |
| 精度 | 要件チェックリスト | ○=1 / ×=0 / 部分的=0.5で合算 |
| 不明瞭点 | subagent自己申告 | 件数と内容（質的改善材料） |
| 裁量補完 | subagent自己申告 | 件数と内容（暗黙仕様の炙り出し） |
| tool_uses | usage meta | シナリオ間の相対値で構造的欠陥を検出 |
| duration_ms | usage meta | 参考値（認知負荷の代替指標） |

### 複数シナリオの並列実行

同一ターン内で複数シナリオのsubagentを並列dispatchする（順次ではなく同時）。

### 結果テーブルフォーマット

Step 2の期待成果物として以下の形式で記録する:

```
| シナリオ | 成功/失敗 | 精度 | 不明瞭点 | 裁量補完 | tool_uses | duration |
|---------|---------|------|---------|---------|----------|---------|
| 中央値   | ○       | 90%  | 1件     | 0件     | 5        | 18s     |
| edge    | ×       | 60%  | 3件     | 2件     | 14       | 41s     |

tool_uses乖離: 14/5 = 2.8倍（閾値3〜5倍以内、構造的欠陥なし）

不明瞭点詳細:
- [edge] [critical]項目が × — [落ちた理由]
- [edge] [具体的な指摘]
```

## dispatch不可の場合

新規subagentをdispatchできない環境（すでにsubagentとして動作中、Task toolが無効など）では:

- このプロトコルを適用しない
- 「dynamic eval skipped: dispatch unavailable」と記録してStep 3（フォーマット準拠チェック）に進む
- 自己再読での代替はバイアスが入るため行わない
- トリガー検証はStep 3（format-checklist.md）のフォールバック項目で代替する
