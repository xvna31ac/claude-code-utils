# doc-audit スキル

ドキュメントを棚卸・校正し、問題点を🔴🟡🟢の3段階で報告する。
修正は承認を得てから行う。

---

## Step 1: 対象ファイルの収集

引数に応じて対象を決める。

- 引数なし → `docs/` 配下の全 `.md` ファイル
- ファイル名指定（例: `design-system-rules.md`） → そのファイルのみ
- フォルダ名指定（例: `design-system/`） → そのフォルダ配下の全 `.md` ファイル

全ファイルをリストアップしてから内容を読む。ファイル数と構成を把握する。

---

## Step 2: 10の観点でチェックする

全ファイルを読んだうえで、以下の観点を順に確認する。
各観点は独立しているが、複数ファイルをまたぐ問題（用語一貫性・冗長性等）は横断的に確認する。

### ⚠️ チェック前の鉄則

**ファイルの存在・場所は必ず Glob ツールで確認してから報告すること。**
「このファイルは誤配置されている」「このパスは存在しない」という🔴報告は、実際に Glob で確認した結果に基づくこと。目視・推測・ドキュメントの記述からの推定での🔴報告は禁止。

**各観点は表面確認にとどまらず深く掘り下げること。**
「書いてある」「書いていない」の確認だけでなく、他ファイルとの整合・実装との一致・エッジケースの未定義・コードの動作可能性まで確認する。チェックリストを埋めることが目的ではなく、実際に問題になりうる箇所を見つけることが目的。

### 観点1: 切り分け妥当性

ドキュメントが正しいフォルダに置かれているかを確認する。

**汎用ドキュメント**（`design-system/` 等）の判断基準：

- このプロジェクト（アプリ）を知らなくても意味が通るか
- 他のプロジェクトでそのまま移植できるか
- 特定のフレームワーク・ツール・アプリ機能に依存していないか

**アプリ固有ドキュメント**（`project/` 等）の判断基準：

- このアプリの機能・ワークフロー・技術スタックに言及しているか
- 「このツール」「このアプリ」という前提で書かれているか

混在している場合：どのセクションが問題かを具体的に指摘する。

---

### 観点2: 参照整合性

ドキュメント内でファイル名・パスを参照している箇所をすべて抽出し、**Glob ツールで実際に存在するか確認する**。

確認方法：

- Markdown リンク `[text](path)` の path
- バッククォートで囲まれたファイル名（例: `` `design-rules.md` ``）
- コードブロック内のファイルパス
- 「〜を参照してください」のような自然言語での参照

**パスの解釈に注意**：ファイル名のみ（`design-system-rules.md`）の記載は、そのファイルが同じディレクトリにあることを示唆する。サブディレクトリ構造がある場合、読者がファイルを見つけられるか確認する。

リネーム・移動後に古いファイル名が残っているケースに特に注意する。

---

### 観点3: 用語一貫性

複数のドキュメントをまたいで同じ概念が異なる表現で書かれていないかを確認する。

チェックポイント：

- 同じ概念の英語/日本語表記が混在（例: "Design System Group" と "デザインシステムグループ"）
- 同じファイル内での表記揺れ（例: "tokens.css" と "token.css"）
- 略語と正式名称の混在
- 固有名詞の大文字小文字のぶれ（例: "Radix UI" と "radix UI"）

---

### 観点4: 構造・順序

各ドキュメントの構成が読みやすいかを確認する。

チェックポイント：

- 最も重要な情報（目的・概要）が冒頭にあるか
- セクションの流れが論理的か（前提 → 詳細 → 例外 等）
- 読み手が知りたい情報の順に並んでいるか
- 見出しレベルの使い方が一貫しているか（H2 → H4 のジャンプ等）

---

### 観点5: 冗長性

複数ドキュメント間での内容の重複、または単一ドキュメント内での繰り返しを確認する。

チェックポイント：

- 同じルール・説明が複数のファイルに書かれていないか
- 「〜については別ドキュメントを参照」と書いてあるが、その内容がこのファイルにも書かれている
- 削除すべき過去のセクションが残っていないか（「旧方式」「廃止予定」等のマーカーがある場合）

完全な重複ではなく「似ているが微妙に違う」場合は🟡として報告する。

---

### 観点6: 可読性

文脈なしで読んで意味が通るかを確認する。

チェックポイント：

- 前提知識なしに読んで意味がわかるか
- 表・コードブロック・箇条書きが適切に使われているか（説明文が長すぎて表にすべき等）
- コードブロックに言語指定があるか（```css,```bash 等）
- 読み手（開発者・AI）を明示してあるか、または自明か

---

### 観点7: 誤字脱字

日本語・英語の明らかな誤りを確認する。

チェックポイント：

- 明らかな変換ミス・誤字
- 英語の技術用語のスペルミス（例: "comonent" → "component"）
- 全角・半角の混在（英数字の全角使用等）
- 句読点の欠落・重複

軽微な指摘が多い場合はサンプルに絞って報告し、残りは「他 N 件類似あり」と記載してよい。

---

### 観点8: 鮮度

実装の変更にドキュメントが追いついていない可能性がある箇所を確認する。

判断のヒント：

- ファイル名・ディレクトリ名が実際の構成と合っているか
- コード例の言語・拡張子が実態と合っているか（例: `.jsx` → `.tsx` 移行後）
- 技術スタックの記載が選定済みの内容と一致しているか
- 「〜は未定」「〜は今後検討」と書いてある項目が実は決定済みでないか

確信が持てない場合は「要確認」として🟡で報告する。

---

### 観点9: カバレッジ欠落

ドキュメントが扱うべき内容を網羅しているかを確認する。

チェックポイント：

- 他のドキュメントで言及されている概念・ルールに対応するドキュメントが存在するか
- `development-guide.md` 等のインデックスに記載されているファイルが実際に存在するか
- フローや手順に「その後どうするか」が書かれていない中断箇所がないか

ドキュメント全体のインデックス（`docs/README.md` 等）がある場合は、そこに記載されたすべてのファイルが存在するかも確認する。

---

### 観点10: 命名一貫性

ファイル名・フォルダ名の命名規則がぶれていないかを確認する。

チェックポイント：

- kebab-case で統一されているか
- プレフィックスの付け方が一貫しているか（例: `design-system-*` と `project-*` の分類）
- README.md の大文字小文字が統一されているか

---

## Step 3: レポートを出力する

以下のフォーマットで報告する。問題がなければ「特になし」と記載する。

```
## ドキュメント棚卸・校正レポート

**対象**: docs/ 配下 N ファイル
**日時**: YYYY-MM-DD

---

### 🔴 要対応
明確な問題。参照切れ・フォルダ誤配置・明らかな整合性の崩れ等。
放置すると次のドキュメント整理作業でコストが上がるもの。

- [ファイル名] 問題の説明

### 🟡 要検討
判断が必要なもの。冗長・鮮度・カバレッジ欠落・「似ているが微妙に違う」等。
修正するかどうかはユーザーが判断する。

- [ファイル名] 問題の説明

### 🟢 軽微
誤字脱字・表記揺れ・命名のぶれ等。対応しなくても実害はないが、直せばきれいになる。

- [ファイル名] L行目: 指摘内容

---

（問題がなければ各セクションに「特になし」と記載）
```

---

## Step 4: 修正フロー

レポート提示後、以下のフローで修正を進める。

1. 「どの項目から対応しますか？」と確認する
2. ユーザーが対応項目を選んだら、修正案を提示する（ファイルは編集しない）
3. ユーザーが承認したらファイルを編集する
4. 次の項目へ

**重要**: 承認なしにファイルを編集してはいけない。
複数ファイルにまたがる変更（リネーム・移動等）は、影響範囲を明示してから確認を取る。
