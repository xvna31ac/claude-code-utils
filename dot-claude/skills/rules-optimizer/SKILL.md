# Rules Optimizer

`.claude/rules/` のモジュラールールシステムを構築・最適化するスキル。

## なぜモジュラールールか

CLAUDE.md は全セッションで読み込まれる。ルールが増えるほどコンテキストを圧迫し、
関係のないルールまで読み込まれる「コンテキスト汚染」が発生する。
`.claude/rules/` の条件付きルールは `paths:` フロントマターで一致したときのみ
読み込まれるため、必要なルールだけを必要なときに適用できる。

## フロー概要

```
Phase 1: 現状分析  →  Phase 2: 設計  →  Phase 3: 生成  →  Phase 4: 検証
```

**モード判定**: Phase 1 の分析結果で以下の2モードを判定する。

- `.claude/rules/` が存在しない → **構築モード**（rules/ を新規作成する）
- `.claude/rules/` が存在する → **最適化モード**（重複排除・paths改善・ルール整理）

---

## Phase 1: 現状分析

### 1-1. 既存ファイルの読み込み

以下を並行して確認する：

1. `CLAUDE.md` — 存在すれば全文読む
2. `.claude/rules/` — 存在すれば全ファイルをリストアップして読む
3. `CLAUDE.local.md` — 存在すれば読む（gitignoreされる個人設定）

### 1-2. プロジェクト特性の把握

設定ファイルを並行して読み、技術スタックを自動検出する：

- `package.json` → フロントエンドフレームワーク・テストツール・言語
- `tsconfig.json` → TypeScript設定・strict mode
- `pyproject.toml` / `setup.py` → Pythonプロジェクト
- `Cargo.toml` → Rustプロジェクト
- `go.mod` → Goプロジェクト
- `docker-compose.yml` / `Dockerfile` → インフラ構成

### 1-3. モード判定と分岐

モード判定の軸は **rules/ の有無のみ**。CLAUDE.md の有無はモード判定に関係しない。

- **構築モード**（rules/ が存在しない）: プロジェクト資材と CLAUDE.md を読んでプロジェクトを理解し、rules/ を新規構築する
- **最適化モード**（rules/ が存在する）: 既存 rules/ とプロジェクト資材を読み、不足ルールの追加・既存ルールの改善・不要ルールの整理を行う

**ヒアリング方針**:
技術スタック・フレームワーク選定はスコープ外のため質問しない。
Claude が知らない**このプロジェクト固有の決定事項**のみを以下の観点で聞く：

- 一般的な慣行から**逸脱・上書きしている**コーディング規約（例：「TSプロジェクトだが DB層はsnake_case」）
- **禁止事項・制約**（例：「`migrations/` は直接触らない」「本番DBへの直接クエリ禁止」）
- **アーキテクチャ上の取り決め**（例：「ロジックはすべてhooksに集約」「Controllerから直接DBを叩かない」）
- **Gitまわりの取り決め**（コミット形式・ブランチ命名等）

ヒアリングの**タイミングは Phase 1-4 の現状レポートと同時に行う**。
Phase 2-2 のベースラインカテゴリ不足もこの時点で一括確認する。往復を最小化するため、
不明点をまとめて1回の質問にすること。

### 1-4. 現状レポート

分析後、以下を簡潔に報告してユーザーの確認を得る：

- モード（構築 / 最適化）
- 検出された技術スタック
- CLAUDE.md の現在の行数（存在する場合）
- 既存 `.claude/rules/` ファイルの一覧（存在する場合）
- 改善・移行候補となるルールの概算数（Phase 2-1 の詳細判定前のため概算）

ヒアリングが必要な場合（構築モードでプロジェクト資材から判断できない情報がある場合、または最適化モードでベースラインカテゴリが未定義の場合）は、このレポートと同時に質問をまとめて行う。

---

## Phase 2: 設計

### 2-1. プロジェクト固有か判定する

**rules に切り出すのはプロジェクト固有の振る舞いルールのみ**。
何も指示しなくても Claude がデフォルトで行う内容は対象外とする。

判定基準：**「このルールを書かなかった場合、Claude は別の動作をするか？」**

| 種別 | 例 | 扱い |
|------|-----|------|
| **対象**（明示しないとClaudeが別の選択をしうる） | 「`migrations/` を直接変更しない」「エラーレスポンスは `{ error: string, code: number }` 形式に統一」「Python ファイルは snake_case（TypeScript 文脈で作業する場合も）」 | rules に切り出す |
| **対象外**（何も言わなくても Claude がそうする） | 「テストを書く」「読みやすいコードを書く」「SQLインジェクションに気をつける」 | rules に含めない |

**用語・概念は rules の対象外**。rules は Claude の振る舞いを制御するものであり、
知識のドキュメントではない。CLAUDE.md に用語定義が含まれていても rules には移行しない。

### 2-2. ベースラインカテゴリの確認

技術スタックによらず、以下のカテゴリは**必ず定義されるべき**。
抽出・ヒアリングの結果にこれらが含まれていない場合、ユーザーに補足を求める。

| カテゴリ | 確認すべき内容 |
|---------|-------------|
| **Gitワークフロー** | コミットメッセージ形式・ブランチ命名・PRの規約・直接pushの制限など |
| **開発フロー上の制約** | 触ってはいけないファイル・実行禁止コマンド・デプロイ手順上の注意など |
| **レビュープロセス** | 作業完了前の自己チェック手順（lint・test・動作確認の順序など） |

未定義のカテゴリがある場合は「このプロジェクトでの取り決めはありますか？」と
具体的に問い合わせる。「特になし」であれば生成をスキップしてよい。

### 2-3. ルールの分類

プロジェクト固有と判定したルールをカテゴリに分類する。
詳細な分類基準は `references/rule-categories.md` を参照。

**最適化モードの場合**: 既存ルールファイルと CLAUDE.md の重複を特定し、
マージ・統合・削除の方針をカテゴリごとに決定する。

### 2-4. paths パターンの決定

技術スタックに応じて `paths:` を設定する。
詳細なパターン例は `references/paths-patterns.md` を参照。

**Gitワークフロー・レビュープロセスの配置方針**:

- プロジェクト固有ルールとして `rules/` に分離するのが基本
- `git-workflow.md`・レビュープロセスファイルともに `paths:` を省略して常時適用とする（特定ファイルに紐付かないため）

**重要**: `paths:` の値は必ずダブルクォートで囲むこと（例: `"src/**/*.ts"`）。
クォートなしだと認識されないバグがある。

### 2-5. 設計の確認

生成・変更予定のファイル一覧と、各ファイルに移行するルールの概要をユーザーに提示する。
承認を得てから Phase 3 へ進む。

## Phase 3: 生成・更新

### 3-1. ルールファイルの生成

`.claude/rules/` 配下にファイルを生成する。

**ファイルフォーマット**:

```markdown
---
paths:
  - "src/components/**/*.tsx"
  - "src/hooks/**/*.ts"
---

# [カテゴリ名]ルール

[移行したルールの内容]
```

**生成の原則**:

- 1ファイル1カテゴリ。関連性の低いルールを混ぜない
- ルールは箇条書きで簡潔に。詳細な説明より明確な指示
- 既存の `.claude/rules/` ファイルと内容が重複する場合はマージする

### 3-2. CLAUDE.md のスリム化

移行したルールを CLAUDE.md から削除する。

**残すもの（全コンテキストで必要なため）**:

- プロジェクト概要（1〜3行）
- 技術スタック一覧
- 開発コマンド（`npm run dev` 等）
- セキュリティ上の絶対禁止事項（`.env` の出力禁止等）
**目標行数**: 150行以内（コンテキスト圧迫を防ぐ実測上の目安）。

### 3-3. サブディレクトリ構成（オプション）

ルールファイルが多く管理しにくい場合はサブディレクトリで整理する：

```
.claude/rules/
├── frontend/
│   ├── react.md
│   └── styles.md
└── backend/
    ├── api.md
    └── database.md
```

---

## Phase 4: 検証

### 4-1. 構文チェック

生成した各ファイルについて：

- YAMLフロントマターの構文が正しいか
- `paths:` の値がダブルクォートで囲まれているか
- Markdownの見出し構造が壊れていないか

### 4-2. 変更サマリーの報告

以下の形式で報告する：

```
## 変更サマリー

### 生成・更新されたルールファイル
- `.claude/rules/code-style.md` — 命名・フォーマット規約（8ルール）
- `.claude/rules/testing.md` — テスト規約（5ルール）
- ...

### CLAUDE.md のスリム化
- 変更前: 287行
- 変更後: 89行（-198行、-69%）

### 適用されるファイルパターン
| ファイル | パターン |
|---------|---------|
| code-style.md | `src/**/*.ts`, `src/**/*.tsx` |
| testing.md | `**/*.test.ts`, `**/*.spec.ts` |
```

**再実行のタイミング**: CLAUDE.md が再び肥大化したとき、
新しいフレームワーク・ツールを導入したとき、ルールが機能していないと感じたときに
このスキルを再実行して最適化する。

---

## エラーハンドリング

- **構築モード** — `.claude/rules/` がない: プロジェクト資材を読んで rules/ を新規構築する（必要に応じてヒアリング）
- **最適化モード** — `.claude/rules/` が存在する: 重複排除・paths改善・不足ルールの追加・不要ルールの整理を行う
- **ルールが少ない（5個未満）**: モジュラール化より CLAUDE.md 整理を優先提案する
- **既存ルールとの競合**: 重複箇所を明示してユーザーに選択させる
