# リファクタ・クリーン

デッドコードを安全に特定・削除する。各ステップでテストによる検証を行う。

## ステップ1: デッドコードを検出する

プロジェクトの種類に応じた分析ツールを実行する:

| ツール | 検出対象 | コマンド |
|--------|---------|---------|
| knip | 未使用のエクスポート・ファイル・依存関係 | `npx knip` |
| depcheck | 未使用のnpm依存関係 | `npx depcheck` |
| ts-prune | 未使用のTypeScriptエクスポート | `npx ts-prune` |
| vulture | 未使用のPythonコード | `vulture src/` |
| deadcode | 未使用のGoコード | `deadcode ./...` |
| cargo-udeps | 未使用のRust依存関係 | `cargo +nightly udeps` |

ツールが利用できない場合は、Grepでインポートがゼロのエクスポートを探す:

```
# エクスポートを見つけ、どこかでインポートされているかを確認する
```

## ステップ2: 所見を分類する

所見を安全度別に分類する:

| 分類 | 例 | 対応 |
|------|---|------|
| **安全** | 未使用のユーティリティ・テストヘルパー・内部関数 | 自信を持って削除 |
| **注意** | コンポーネント・APIルート・ミドルウェア | 動的インポートや外部利用者がないか確認 |
| **危険** | 設定ファイル・エントリポイント・型定義 | 触る前に調査 |

## ステップ3: 安全な削除ループ

各**安全**アイテムについて:

1. **全テストスイートを実行** — ベースラインを確立する（全グリーン）
2. **デッドコードを削除** — Editツールで的確に削除する
3. **テストスイートを再実行** — 何も壊れていないことを確認する
4. **テストが失敗した場合** — 即座に `git checkout -- <file>` で元に戻し、このアイテムをスキップ
5. **テストが通過した場合** — 次のアイテムへ進む

## ステップ4: 注意アイテムを処理する

**注意**アイテムを削除する前に:

- 動的インポートを検索: `import()`・`require()`・`__import__`
- 文字列参照を検索: 設定内のルート名・コンポーネント名
- 公開パッケージAPIからエクスポートされているか確認する
- 外部利用者がいないか確認する（公開パッケージの場合は依存元を確認）

## ステップ5: 重複を統合する

デッドコード削除後に探すもの:

- 準重複関数（80%以上類似）— 一つにマージする
- 冗長な型定義 — 統合する
- 価値を追加しないラッパー関数 — インライン化する
- 目的を持たない再エクスポート — 間接参照を削除する

## ステップ6: サマリー

結果を報告する:

```
デッドコードクリーンアップ
──────────────────────────────
削除済み: 未使用関数 12件
          未使用ファイル 3件
          未使用依存関係 5件
スキップ:  2件（テスト失敗）
削減量:    約450行削除
──────────────────────────────
全テスト通過 ✅
```

## ルール

- **テストを実行せずに削除しない**
- **一度に一つの削除** — アトミックな変更でロールバックを容易にする
- **不確かな場合はスキップ** — 本番を壊すよりデッドコードを残す方が良い
- **クリーニング中にリファクタリングしない** — 関心を分離する（まずクリーン、後でリファクタリング）
