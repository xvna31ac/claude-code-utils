# ビルドと修正

ビルドエラーと型エラーを、最小限で安全な変更で段階的に修正する。

## ステップ1: ビルドシステムを検出する

プロジェクトのビルドツールを特定し、ビルドを実行する:

| 指標 | ビルドコマンド |
|------|--------------|
| `build`スクリプトを含む`package.json` | `npm run build` または `pnpm build` |
| `tsconfig.json`（TypeScriptのみ） | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build 2>&1` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m py_compile` または `mypy .` |

## ステップ2: エラーを解析してグループ化する

1. ビルドコマンドを実行してstderrをキャプチャする
2. エラーをファイルパスごとにグループ化する
3. 依存関係の順序でソートする（ロジックエラーより前にインポート/型を修正）
4. 進捗追跡のために総エラー数を数える

## ステップ3: 修正ループ（一度に一つのエラー）

各エラーについて:

1. **ファイルを読む** — Readツールでエラーのコンテキストを確認する（エラーの前後10行）
2. **診断する** — 根本原因を特定する（インポート不足・型の誤り・構文エラー）
3. **最小限に修正する** — Editツールでエラーを解消する最小限の変更を行う
4. **ビルドを再実行する** — エラーが解消され、新しいエラーが発生していないことを確認する
5. **次へ進む** — 残りのエラーを続けて対処する

## ステップ4: ガードレール

以下の場合はユーザーに確認する:

- 修正が**解消するより多くのエラーを発生させる**場合
- **3回試みても同じエラーが続く**場合（より深い問題の可能性）
- 修正に**アーキテクチャの変更が必要**な場合（ビルド修正ではない）
- ビルドエラーが**依存関係の欠如**に起因する場合（`npm install`・`cargo add`等が必要）

## ステップ5: サマリー

結果を表示する:

- 修正されたエラー（ファイルパスとともに）
- 残っているエラー（あれば）
- 新たに発生したエラー（ゼロであるべき）
- 未解決の問題に対する推奨される次のステップ

## リカバリー戦略

| 状況 | 対応 |
|------|------|
| モジュール/インポートが見つからない | パッケージがインストールされているか確認し、インストールコマンドを提案する |
| 型の不一致 | 両方の型定義を読み、より狭い型を修正する |
| 循環依存 | インポートグラフでサイクルを特定し、抽出を提案する |
| バージョン競合 | `package.json`/`Cargo.toml`のバージョン制約を確認する |
| ビルドツールの設定ミス | 設定ファイルを読み、動作するデフォルトと比較する |

安全のために一度に一つのエラーを修正する。リファクタリングより最小限の差分を優先する。
