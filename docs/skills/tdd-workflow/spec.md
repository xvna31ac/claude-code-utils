新機能・バグ修正・リファクタリング等のコード開発に対してTDD原則を強制し、80%以上のカバレッジを確保するスキル tdd-workflow を以下の仕様で作成してください。

# ファイル構成

- SKILL.mdのみ（references/・templates/・scripts/ なし）

# フロー

- Step 1: ユーザージャーニーを書く（role/action/benefit形式）
- Step 2: テストケースを生成する（各ユーザージャーニーに対して包括的なテストケースを作成）
- Step 3: テストを実行する（失敗することを確認する。まだ実装していないため）
- Step 4: コードを実装する（テストをパスするための最小実装を書く）
- Step 5: テストを再実行する（パスすることを確認する）
- Step 6: リファクタリング（テストをグリーンに保ちながらコード品質を改善する）
- Step 7: カバレッジを確認する（80%以上達成を検証する）

# 制御フロー

- 明示的な分岐・ループ・フォールバックの記述なし
- TDDサイクル（テスト作成→失敗確認→実装→パス確認）は各Stepで暗黙的に繰り返す

# 出力の完全性

## テストファイル組織構造

以下のディレクトリ構造でテストファイルを配置する:

src/
- components/[コンポーネント名]/[コンポーネント名].test.tsx （ユニットテスト）
- app/api/[エンドポイント]/route.test.ts （インテグレーションテスト）

e2e/
- [機能名].spec.ts （E2Eテスト）

## ファイル保存・チャット出力

- 明示的なファイル保存指示なし
- チャット出力の明示的な定義なし

# 承認ゲート

- 明示的な承認ゲートなし

# 評価・判定ロジック

## カバレッジ成功基準

- 最低80%のカバレッジ（ユニット + インテグレーション + E2E）
- カバレッジ閾値は以下4軸すべてで80%:
  - branches: 80
  - functions: 80
  - lines: 80
  - statements: 80

## 成功メトリクス（全て満たすこと）

- 80%以上のコードカバレッジ達成
- 全テストがパス（グリーン）
- スキップまたは無効化されたテストがない
- ユニットテストの実行が30秒未満
- E2Eテストが重要なユーザーフローをカバーしている
- テストが本番前のバグを検出できている

# チェックリスト型の項目

## Best Practices（10項目）

1. Write Tests First: 常にTDDで行う
2. One Assert Per Test: 単一の振る舞いにフォーカスする
3. Descriptive Test Names: 何をテストしているかを説明する
4. Arrange-Act-Assert: 明確なテスト構造を使う
5. Mock External Dependencies: ユニットテストを分離する
6. Test Edge Cases: null・undefined・空・大きな値をカバーする
7. Test Error Paths: ハッピーパスだけでなくエラーパスもテストする
8. Keep Tests Fast: ユニットテストは1件50ms未満
9. Clean Up After Tests: 副作用を残さない
10. Review Coverage Reports: ギャップを特定する

## Success Metrics（6項目）

- 80%以上のコードカバレッジ達成
- 全テストがパス（グリーン）
- スキップまたは無効化されたテストがない
- ユニットテストの実行が30秒未満
- E2Eテストが重要なユーザーフローをカバーしている
- テストが本番前のバグを検出できている

## Common Testing Mistakes to Avoid（3セット）

パターン1: 実装詳細のテスト
- NG: 内部状態（component.state.count等）を直接テストする
- 正しい: ユーザーに見えるUI上の振る舞いをテストする

パターン2: 壊れやすいセレクター
- NG: CSSクラス（.css-class-xyz等）でクリックする
- 正しい: セマンティックセレクター（button:has-text やdata-testid属性）を使う

パターン3: テストの非独立性
- NG: テストが互いに依存している（前のテストの結果に依存する）
- 正しい: 各テストが自分でデータをセットアップして独立している

# 優先順位

- Tests BEFORE Code: 常にテストを先に書き、次にコードを実装する（最優先原則）
- テスト種別の適用優先: ユニットテスト → インテグレーションテスト → E2Eテストの順で記述する

# 散文埋め込みの判断ルール

## テスト種別の適用基準

各テストタイプをいつ使うかの判断基準:

- ユニットテスト: 個別の関数・ユーティリティ・コンポーネントロジック・純粋関数・ヘルパーに対して使う
- インテグレーションテスト: APIエンドポイント・データベース操作・サービス間連携・外部API呼び出しに対して使う
- E2Eテスト（Playwright）: 重要なユーザーフロー・完全なワークフロー・ブラウザ操作・UI操作に対して使う

## TDD原則

- テストを先に書く（ALWAYS）。コードを先に書いてからテストを後付けしてはいけない
- リファクタリング時はテストをグリーンに保ち続けること
- 重複の除去・命名の改善・パフォーマンス最適化・可読性向上をリファクタリングの目標とする

# 適用除外条件・NGパターン

## 適用する場面

- 新機能または機能の追加
- バグや問題の修正
- 既存コードのリファクタリング
- APIエンドポイントの追加
- 新しいコンポーネントの作成

## NGパターン（Common Testing Mistakes）

- 実装詳細（内部状態・内部メソッド）をテストする: NG
- CSSクラスや不安定なセレクターを使う: NG（セマンティックセレクターまたはdata-testid属性を使うこと）
- テスト間の依存関係を持たせる: NG（各テストは独立させること）
- テストをスキップ・無効化する: NG
- 後からチェックリストに[critical]タグを付け外しする: NG（仕様は事前に固定する）
