# 評価・判定ロジック

## 動的eval（3部構成）

Part 0: 設計整合性
- 新規subagentにSKILL.md全体と全補助ファイルを白紙で読ませる
- 評価観点: 用語の一貫性、哲学的整合性、根拠不明な要素、セクション間矛盾
- 判定: 問題1件でも検出 → 「設計整合性: 要修正N件」→ Step 5改善計画で最優先
- 判定: 問題なし → 「設計整合性: 問題なし」と記録してPart 1へ

Part 1: frontmatter/トリガー検証
- 競合スキルを2から3件特定してdescriptionを収集する
- 正例（対象スキルが選ばれるべき発言）2から3件、負例（競合スキルが正解の発言）1から2件を設計する
- 新規subagentにスキルのdescription一覧と各発言を渡して「どのスキルを呼ぶか」を判定させる
- 正例で対象スキルが選ばれた: OK
- 正例で別スキルが選ばれた（アンダートリガー）: descriptionの1文目・トリガー例を修正
- 負例で対象スキルが選ばれた（オーバートリガー）: トリガー条件を絞る

Part 2: body/実行検証
- シナリオ: 中央値1件（最も典型的なユースケース）+ edge 1から2件（境界条件・イレギュラーな入力）
- シナリオは事前に固定する（結果を見てから変えない）
- 異なる使用文脈から導出する: 典型的な利用者 + 想定外の利用者（初めて使う・ドメイン知識が薄い）の少なくとも2視点を含める
- 新規subagentへの指示: SKILL.mdのパスをReadして把握し、シナリオを実行して、成果物・要件達成・不明瞭点・裁量補完・再試行回数を返す
- criticalタグ付き要件が全て○のみ成功（1つでも×または部分的は失敗）
- 精度 = 満たした項目数 / 全項目数（○=1、×=0、部分的=0.5）
- tool_uses: シナリオ間で3から5倍以上乖離した場合 → references/への過度な依存として記録

動的evalと静的スコアが矛盾する場合: 動的evalを優先。Step 5の改善計画に「⚠️ 動的eval優先」と明記

同一agentを使い回さない: Part 0・Part 1・各Part 2シナリオそれぞれで新規subagentをdispatch

## 静的スコア（5軸100点）

軸1: 専門知識網羅性 30点
軸2: 柔軟性・汎用性 25点
軸3: 実用価値 20点
軸4: 保守性 15点
軸5: 補助資材 10点

グレード: S=90から100 / A=80から89 / B=70から79 / C=60から69 / D=59以下

## Step 7 完了条件

主: 動的evalの不明瞭点・裁量補完が解消されている
副: その結果としてのグレードが確認されている

## Step 7 設計整合性の再チェック条件（以下のいずれかに該当する場合）

- 新機能・新セクションを追加した改善がある
- Step 5→Step 7のループが2回以上に及んでいる（累積的なつぎはぎリスク）

# 優先順位

評価層: 動的eval > フォーマット準拠 > 静的品質スコア
改善計画: 設計整合性問題 → 動的evalの不明瞭点・裁量補完 → Criticalフォーマット違反 → 静的スコア低軸
Step 6の実施順: 設計整合性問題から導かれた改善 → 動的eval不明瞭点・裁量補完から導かれた改善

スコアの性質判定:
- 「妥当なX」: 残りギャップを埋める改善がスコア最適化のみ（実用価値を伴わない）
- 「改善余地あるX」: 実用的な改善が存在し、それを実施すれば自然に上のグレードに達する

# 散文埋め込みの判断ルール

改善案の整合性確認（Step 5）
- 改善案を提示する前に「この改善案はスキルの既存設計哲学と整合するか」を確認
- 矛盾する場合は「⚠️設計哲学と要確認」を付記

修正後の自己確認（Step 6）
- 「この修正が改善計画のどの課題を解消するか」を自己確認
- 新たな問題を導入していないかを確認

新機能追加時の確認（Step 6）
- 「この要素はスキルの設計思想の文脈で正しい位置にあるか（静的に置くべきか動的か、どのファイルに属するか）」を明示的に確認

# スキルタイプ別の動作差異（Part 2シナリオ）

通常スキル（technique・reference）: 典型ユースケース + edge case
discipline-enforcing（TDD・セキュリティ強制など）: 圧力シナリオを使用

圧力シナリオの構成: 以下のうち3つ以上を組み合わせてエージェントがルールを破りたくなる状況を作る
- 時間（緊急デプロイ、締め切り5分前）
- サンクコスト（3時間かけた実装）
- 権限（上司が「今すぐやれ」と言っている）
- 疲労（終業時刻、夕食の予定がある）
- 経済的（本番障害で$10k/分の損失）

# チェックリスト型

## format-checklist.md（チェックリスト型）

frontmatter:
- nameがケバブケースか（Critical）
- descriptionが存在するか（Critical）
- descriptionが|ブロックスカラーか（>は改行がスペースになるため不可）（Important）
- descriptionが3文構成か（1文目:動詞で始まる役割説明 / 2文目:「〜が必要な場合に自動適用されます」 / 3文目:トリガー例）（Important）
- descriptionがワークフロー要約になっていないか（要約があるとClaudeがbodyを読まずにdescriptionの手順だけを実行してしまう）（Important）
- toolsフィールドがあれば削除を提案（Nice to have）

body構造:
- 各Stepが「実行内容/専門性/期待成果物/完了基準」を含むか（Step 1が引数あり/なし分岐形式の場合は対象外）（Important）
- body内に「適用条件」「使用場面」「When to Use」セクションがないか（Important）
- SKILL.mdが500行以内か（超過する場合はreferences/への分離を提案）（Important）

Progressive Disclosure:
- 冒頭での一括Readがないか（必要なStepで明示的にReadするのが正しい）（Important）
- SKILL.mdとreferences/の重複がないか（Important）

トリガー精度:
（dynamic eval実施済みの場合: シナリオ結果を優先しこのセクションスキップ可）
（dynamic eval skippedの場合は全項目実施）
- 代表発言での発火確認（競合スキル2から3件のdescriptionを参照し正例2から3件・負例1から2件で判定）（Important）
- 250文字以内のキーワード配置（Important）

参照パスの正確性:
- 完全パス（プロジェクトルート相対）か（references/foo.mdは不可、.claude/skills/xxx/references/foo.mdが正しい）（Critical）
- ワイルドカードを使っていないか（Critical）
- 参照先ファイルが実際に存在するか（Globで確認）（Critical）

参照整合性:
- テンプレートと生成例の一致（references/が存在する場合）（Important）
- スキル間依存の存在確認 - 呼び出し先が.claude/skills/に実在するか（他スキル呼び出しがある場合）（Important）
- 循環参照がないか（他スキル呼び出しがある場合）（Important）
- CLIツールが存在するか（which コマンドで確認）（CLIコマンドの使用がある場合）（Important）
- 依存ディレクトリが存在するか（ディレクトリ依存がある場合）（Nice to have）
- 300行超のreferences/ファイルにTOCがあるか（references/が存在する場合）（Nice to have）

意味的整合性:
- 承認ゲートの配置（破壊的変更前・設計確定の場面にあるか。読み取り・分析のみのフェーズに不要なゲートがないか）（Important）
- 条件分岐の完全性（条件付きフェーズに「条件を満たさない場合の動作」が明示されているか）（Important）

自己フィードバック（各指摘をレポートに含める前に確認）:
- False Positive排除: 指摘がファイル内に実際に存在するかを再確認（存在しない問題の指摘は除外）
- 重大度の妥当性: 「動作を壊すか」を基準にCriticalを厳選
- 修正案の具体性: 「〜を〜に変更する」形式で記載（「〜を検討する」は不可）

## static-scoring-rubric.md（採点基準）

軸1 専門知識網羅性（30点）:
- 25から30: 業界標準の主要項目を90%以上カバー、最新バージョン準拠
- 18から24: 70から89%カバー、一部の重要な観点が欠落
- 10から17: 50から69%カバー、複数の重要な観点が欠落
- 0から9: 50%未満、専門知識がほぼない
- 高頻度更新ドメイン（security・AI/ML等）: WebSearchで最新標準を確認してから採点
- プロセス・メタスキルの採点: 手法の選択理由の明示・代替アプローチへの言及・理論的背景の正確な参照で評価

軸2 柔軟性・汎用性（25点）:
- 21から25: 特定例への言及が最小限、edge caseの扱いが明示
- 15から20: 概ね汎用的だが特定ツール・フレームワーク依存がある
- 8から14: 特定の例やプロジェクトを前提とした記述が多い
- 0から7: 特定ユースケースにしか通用しない

軸3 実用価値（20点）:
- 17から20: Before/Afterが明確、30分以上の作業を大幅自動化
- 11から16: 一定の業務削減効果、一部手動作業が残る
- 5から10: 効果は限定的
- 0から4: 業務削減効果がほぼない

軸4 保守性（15点）:
- 13から15: DRY原則徹底、SKILL.mdとreferences/に重複なし、変更箇所局所的
- 8から12: 概ねDRYだが一部重複
- 4から7: 重複が多い、1つの変更が複数箇所に影響
- 0から3: 同じ情報があちこちに書かれている
- トークン効率も評価する: 頻繁使用スキルは200行以内、その他は500行以内

軸5 補助資材（10点）:
- 9から10: 必要な補助資材が揃い、必要なStepで明示的にRead、300行超に目次あり
- 6から8: 補助資材はあるが一部欠けまたは参照方法が不適切
- 3から5: 補助資材が最小限
- 0から2: 補助資材がほぼなし、または冒頭で一括Read（Progressive Disclosure違反）

スコアの性質は評価レポートとreport-templateの両方に必ず明示する:
- 妥当なX: 残りギャップを埋める改善がスコア最適化のみ（実用価値を伴わない）
- 改善余地あるX: 実用的な改善が存在し、それを実施すれば自然に上のグレードに達する
